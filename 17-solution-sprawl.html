Solution Sprawl

Description

The code that implements one cohesive solution — one feature, one capability, one design decision — is spread thinly across many classes, with no single place you can point to and say "this is how we do X." To understand the solution, you have to trace through a chain of classes, piecing together fragments. To change the solution, you have to find and update every fragment.

Solution Sprawl is close to Shotgun Surgery, but the emphasis is different. Shotgun Surgery is felt when you change something and discover you need to edit many files. Solution Sprawl is felt when you read the code and can't find the solution — it's nowhere and everywhere at once. Shotgun Surgery is the pain of modification; Solution Sprawl is the pain of comprehension.

Kerievsky identifies this specifically as a pattern where creational logic — how objects are constructed, configured, and wired together — gets smeared across multiple classes instead of being concentrated in one factory or builder.

Example

Suppose the system needs to create a database connection with specific configuration. The logic is sprawled across four classes:

public class AppConfig {
    public String getDbUrl() {
        return System.getenv("DB_URL");   // piece 1: reads the URL
    }
}

public class ConnectionProperties {
    public Properties buildProperties() {
        Properties props = new Properties();
        props.put("user", System.getenv("DB_USER"));        // piece 2: reads credentials
        props.put("password", System.getenv("DB_PASSWORD"));
        props.put("ssl", "true");
        return props;
    }
}

public class PoolConfig {
    public int getMaxConnections() { return 10; }    // piece 3: pool settings
    public int getTimeout() { return 5000; }
    public boolean isValidationEnabled() { return true; }
}

public class DatabaseService {
    public Connection connect() {
        AppConfig config = new AppConfig();               // uses piece 1
        ConnectionProperties connProps = new ConnectionProperties(); // uses piece 2
        PoolConfig poolConfig = new PoolConfig();         // uses piece 3

        String url = config.getDbUrl();
        Properties props = connProps.buildProperties();

        // piece 4: actually creates the connection with pool settings
        DataSource ds = createPooledDataSource(url, props,
            poolConfig.getMaxConnections(),
            poolConfig.getTimeout(),
            poolConfig.isValidationEnabled());
        return ds.getConnection();
    }

    }
}

The "solution" to "how we connect to the database" is spread across AppConfig (URL), ConnectionProperties (credentials), PoolConfig (pool settings), and DatabaseService (assembly). None of these classes tells the full story. To understand or change how connections work, you have to visit all four and mentally reassemble the pieces. If the company switches from environment variables to a secrets vault, you're editing three classes.

What principle does it break?

Cohesion. The pieces of one solution belong together. When they're scattered, no single class is responsible for the solution, which means no single class can be understood, tested, or changed in isolation. The knowledge of "how we connect to the database" is distributed, which means nobody owns it.

This also violates Separation of Concerns in a subtle way: the concern is properly separated from other concerns (database connection is separate from order processing), but it's not properly gathered — its own internal parts are dispersed.

How to fix it

Condense the sprawl into a single class

Pull all the fragments into one cohesive class that owns the entire solution:

public class DatabaseConnectionFactory {
    private final String url;
    private final String user;
    private final String password;
    private final int maxConnections;
    private final int timeoutMs;
    private final boolean validateConnections;

    public DatabaseConnectionFactory() {
        this.url = System.getenv("DB_URL");
        this.user = System.getenv("DB_USER");
        this.password = System.getenv("DB_PASSWORD");
        this.maxConnections = 10;
        this.timeoutMs = 5000;
        this.validateConnections = true;
    }

    public DatabaseConnectionFactory(String url, String user, String password,
                                      int maxConnections, int timeoutMs,
                                      boolean validateConnections) {
        this.url = url;
        this.user = user;
        this.password = password;
        this.maxConnections = maxConnections;
        this.timeoutMs = timeoutMs;
        this.validateConnections = validateConnections;
    }

    public Connection getConnection() {
        Properties props = new Properties();
        props.put("user", user);
        props.put("password", password);
        props.put("ssl", "true");

        DataSource ds = createPooledDataSource(props);
        return ds.getConnection();
    }

    private DataSource createPooledDataSource(Properties props) {
        // configures pool with url, props, maxConnections, timeoutMs, validateConnections
        return null;
    }
}

One class, one solution, one place to look. The three fragment classes (AppConfig, ConnectionProperties, PoolConfig) can be removed (or, if they serve other purposes, their database-specific parts are extracted into this factory).

Now "how do we connect to the database?" has exactly one answer: DatabaseConnectionFactory.

Apply a Creation Method or Factory

Kerievsky specifically recommends replacing sprawled construction logic with a Creation Method — a single, intention-revealing method that encapsulates the full creation process:

public class DatabaseConnectionFactory {

    public static Connection createDefault() {
        return new DatabaseConnectionFactory().getConnection();
    }

    public static Connection createForTesting() {
        return new DatabaseConnectionFactory(
            "jdbc:h2:mem:test", "sa", "", 2, 1000, false
        ).getConnection();
    }

    // ... full class as above
}

Every caller uses DatabaseConnectionFactory.createDefault() or DatabaseConnectionFactory.createForTesting(). The sprawl is gone — creation is consolidated, and the intent is named.
