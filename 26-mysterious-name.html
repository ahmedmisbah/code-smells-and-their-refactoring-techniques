Mysterious Name

Also known as: Inefficient Name

Description

A variable, method, class, or parameter whose name doesn't communicate its purpose. It might be a single letter (x, d, tmp), a cryptic abbreviation (calcPrcDscPct), a misleading name (a method called process that validates), or a name so generic it could mean anything (data, info, handle, manager, doStuff).

Naming is the most frequent act of design in programming. A good name replaces a comment, prevents misuse, and lets a reader understand code without tracing through its implementation. A bad name forces the reader to reverse-engineer meaning from context — or worse, builds a false mental model that leads to bugs.

Fowler elevated this to a first-class smell in his second edition because poor naming is the single most common barrier to code comprehension.

Example

public class Proc {
    private List<Map<String, Object>> d;
    private int st;

    public double calc(Map<String, Object> r) {
        double t = 0;
        for (Map<String, Object> e : d) {
            double v = ((Number) e.get("a")).doubleValue();
            int q = ((Number) e.get("b")).intValue();
            double lp = v * q;
            if ((int) e.get("c") == 1) {
                lp *= 0.9;
            }
            t += lp;
        }
        st = 1;
        return t;
    }
}

Without reading every line and decoding the single-letter names, this code is opaque. What is Proc? What does d hold? What do a, b, c in the map represent? What does st track? What does calc calculate? Every name requires the reader to solve a puzzle.

What principle does it break?

Readability and intent-revealing design. Code is read far more often than it's written. A name is the primary tool for communicating intent to the next reader (including your future self). Mysterious names force readers to hold context in their head, slow comprehension, and increase the risk of misunderstanding.

How to fix it

Rename Variable, Rename Method, Rename Class

The fix is straightforward: give things names that reveal their purpose.

public class OrderPricingEngine {
    private List<OrderItem> items;
    private boolean calculated;

    public double calculateTotal(Order order) {
        double total = 0;
        for (OrderItem item : items) {
            double unitPrice = item.getPrice();
            int quantity = item.getQuantity();
            double linePrice = unitPrice * quantity;
            if (item.isDiscountEligible()) {
                linePrice *= 0.9;
            }
            total += linePrice;
        }
        calculated = true;
        return total;
    }
}

Same logic, zero puzzles. Every name answers the question "what is this?"

Naming guidelines

Methods: verb + noun describing what it does — calculateTotal, findUserById, validateOrder. If you can't name it clearly, it's probably doing too many things (see Long Method).
Variables: describe the contents — unitPrice not v, lineItems not d, retryCount not rc.
Classes: noun describing the concept — OrderPricingEngine not Proc, CustomerRepository not CustMgr.
Booleans: phrase as a question — isValid, hasPermission, canRetry.
Avoid generic names: data, info, item, result, temp, thing only when the context genuinely is generic (like a utility method that operates on any type).
