Refused Bequest

Description

A subclass inherits methods or fields from a parent but doesn't use them — or worse, actively overrides them to do nothing or throw exceptions. The child takes the inheritance but refuses the bequest. The parent says "you can do X, Y, and Z" and the child says "actually, I don't do Z — please ignore that."

This usually means the inheritance hierarchy is wrong. The subclass isn't really a special case of the parent — it just happens to share some code. Inheritance was used for code reuse when composition would have been more honest.

The classic example: a Stack that extends ArrayList. A stack isn't a list — it doesn't support add(index, element) or get(index) — but it inherits those methods. The inherited interface is a lie.

Example

public class Bird {
    private String name;

    public String getName() { return name; }

    public void fly() {
        System.out.println(name + " is flying");
    }

    public void layEggs() {
        System.out.println(name + " laid eggs");
    }

    public double getAltitude() {
        return 100.0; // default cruising altitude
    }
}

public class Penguin extends Bird {

    @Override
    public void fly() {
        throw new UnsupportedOperationException("Penguins can't fly");
    }

    @Override
    public double getAltitude() {
        return 0; // meaningless — but forced to implement
    }

    public void swim() {
        System.out.println(getName() + " is swimming");
    }
}

Penguin inherits fly() and getAltitude() but refuses both — one throws, the other returns a meaningless value. Any code that receives a Bird and calls fly() will blow up if it gets a Penguin. The contract of the parent is violated.

public class BirdTracker {
    public void trackMigration(List<Bird> flock) {
        for (Bird bird : flock) {
            bird.fly(); // crashes on Penguins
            double alt = bird.getAltitude();
            logPosition(bird.getName(), alt);
        }
    }
}

What principle does it break?

Liskov Substitution Principle (LSP). A subclass should be usable wherever its parent is expected without breaking behavior. If calling fly() on a Bird can throw UnsupportedOperationException, then Bird is not a reliable abstraction — you can't trust its contract. Code that depends on Bird must now check the concrete type before calling methods, which defeats the purpose of polymorphism.

It also signals a misuse of inheritance: the relationship is "reuses some code from" rather than "is a special kind of."

Personal note — Interface Segregation Principle (ISP). I also consider Refused Bequest a violation of ISP, viewed from a slightly different angle. The parent type (Bird) is a fat interface that bundles capabilities not all subclasses can support. Clients like BirdTracker are forced to depend on the full Bird type — including fly(), getAltitude(), layEggs(), and everything else — even though they only care about flight. The subclass (Penguin) is forced to carry methods it can't fulfill (LSP pain on the implementer side), while the client is forced to depend on methods it doesn't need (ISP pain on the consumer side). The two violations share the same root cause — an overly broad parent type — and the fix is the same: break it into focused interfaces (Flyable, Swimmable), which solves both problems simultaneously.

How to fix it

Replace Inheritance with Delegation (Composition)

If the subclass only wants some of the parent's behavior, don't inherit — compose.

public class Bird {
    private final String name;

    public Bird(String name) { this.name = name; }
    public String getName() { return name; }
    public void layEggs() {
        System.out.println(name + " laid eggs");
    }
}

public interface Flyable {
    void fly();
    double getAltitude();
}

public interface Swimmable {
    void swim();
}

public class Eagle extends Bird implements Flyable {
    public Eagle(String name) { super(name); }

    @Override
    public void fly() {
        System.out.println(getName() + " is soaring");
    }

    @Override
    public double getAltitude() { return 300.0; }
}

public class Penguin extends Bird implements Swimmable {
    public Penguin(String name) { super(name); }

    @Override
    public void swim() {
        System.out.println(getName() + " is swimming");
    }
}

Penguin no longer inherits fly(). The tracker works with Flyable, not Bird:

public class BirdTracker {
    public void trackMigration(List<Flyable> flock) {
        for (Flyable bird : flock) {
            bird.fly();  // only Flyable birds — safe
            double alt = bird.getAltitude();
            logPosition(alt);
        }
    }
}

A Penguin can't be passed to trackMigration — the type system prevents the mistake at compile time.

Push Down Method / Push Down Field

When only some subclasses use a method or field defined in the parent, push it down to the subclasses that actually need it.

Before — parent defines behavior only some children use:

public abstract class Vehicle {
    public abstract void startEngine();
    public abstract void steer();
    public abstract void pedal(); // only applies to bicycles
}

public class Car extends Vehicle {
    public void startEngine() { /* ... */ }
    public void steer() { /* ... */ }
    public void pedal() {
        throw new UnsupportedOperationException(); // refused bequest
    }
}

public class Bicycle extends Vehicle {
    public void startEngine() {
        throw new UnsupportedOperationException(); // refused bequest
    }
    public void steer() { /* ... */ }
    public void pedal() { /* ... */ }
}

After — push specialized methods down, keep only shared behavior in the parent:

public abstract class Vehicle {
    public abstract void steer();
}

public class Car extends Vehicle {
    public void startEngine() { /* ... */ }
    @Override
    public void steer() { /* ... */ }
}

public class Bicycle extends Vehicle {
    public void pedal() { /* ... */ }
    @Override
    public void steer() { /* ... */ }
}

The parent only contains what all children genuinely share. No child refuses anything.
