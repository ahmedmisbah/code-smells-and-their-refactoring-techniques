Data Clumps

Description

The same group of fields or parameters appears together in multiple places — different classes, different method signatures, different data structures — but never gets recognized as a concept of its own. Three fields that always travel as a pack (street, city, zipCode) are a clump. Two parameters that show up side by side in five method signatures (startDate, endDate) are a clump. The data is clearly related, but nobody has given the relationship a name.

Data Clumps are close relatives of Long Parameter List and Primitive Obsession. The difference in emphasis is this: Long Parameter List focuses on the pain at the call site; Primitive Obsession focuses on the missing type; Data Clumps focuses on the repetition — the same group copy-pasted across the codebase. Where there's a clump, there's almost always hidden behavior waiting to be discovered once you give it a home.

A useful litmus test: if you removed one member of the group, would the remaining members still make sense? If not, they belong together in a class.

Example

public class Customer {
    private String name;
    private String street;
    private String city;
    private String state;
    private String zipCode;

    public String formatAddress() {
        return street + ", " + city + ", " + state + " " + zipCode;
    }
}

public class Supplier {
    private String companyName;
    private String street;
    private String city;
    private String state;
    private String zipCode;

    public String formatAddress() {
        return street + ", " + city + ", " + state + " " + zipCode;
    }
}

public class Warehouse {
    private String warehouseId;
    private String street;
    private String city;
    private String state;
    private String zipCode;

    public double distanceTo(String otherStreet, String otherCity,
                              String otherState, String otherZipCode) {
        // geocode both addresses and compute distance
        return 0;
    }
}

public class ShippingService {
    public double calculateRate(String fromStreet, String fromCity,
                                 String fromState, String fromZipCode,
                                 String toStreet, String toCity,
                                 String toState, String toZipCode,
                                 double weightKg) {
        // ... nine parameters, eight of which are two address clumps
        return 0;
    }
}

The four-field address group (street, city, state, zipCode) appears in Customer, Supplier, Warehouse, and twice in ShippingService.calculateRate. The formatAddress logic is duplicated. The distanceTo method takes a second address as four loose strings. The shipping method takes two addresses as eight strings — a nine-parameter signature where a single swap silently corrupts the result.

What principle does it break?

DRY (Don't Repeat Yourself). The same cluster of fields and the same logic operating on them is duplicated across every class and method that needs an address. When the address format changes (say, adding a country field), you have to hunt down every clump and update it individually — a textbook Shotgun Surgery situation.

It also violates cohesion: each host class carries fields that aren't really about its core concept. A Customer isn't an address; it has an address. Mixing the two reduces clarity about what each class is actually responsible for.

How to fix it

The primary refactoring is Extract Class: pull the clump into its own class and replace all occurrences.

public class Address {
    private final String street;
    private final String city;
    private final String state;
    private final String zipCode;

    public Address(String street, String city, String state, String zipCode) {
        this.street = street;
        this.city = city;
        this.state = state;
        this.zipCode = zipCode;
    }

    public String format() {
        return street + ", " + city + ", " + state + " " + zipCode;
    }

    public double distanceTo(Address other) {
        // geocode both and compute distance — lives here now, once
        return 0;
    }

    // getters, equals, hashCode...
}

Every host class now delegates to Address:

public class Customer {
    private String name;
    private Address address;

    public String formatAddress() {
        return address.format();
    }
}

public class Supplier {
    private String companyName;
    private Address address;

    public String formatAddress() {
        return address.format();
    }
}

public class Warehouse {
    private String warehouseId;
    private Address address;

    public double distanceTo(Warehouse other) {
        return address.distanceTo(other.address);
    }
}

public class ShippingService {
    public double calculateRate(Address from, Address to, double weightKg) {
        double distance = from.distanceTo(to);
        // ... three parameters instead of nine
        return 0;
    }
}

The nine-parameter calculateRate becomes three. The duplicated formatAddress logic now lives in one place. And distanceTo naturally belongs to Address, where it's written once and shared everywhere.

Supporting refactorings

Introduce Parameter Object

This is Extract Class applied specifically to method signatures. When the same group of parameters recurs across methods, wrap them.

Before — the same three parameters appear in every reporting method:

public class Analytics {
    public Report dailyReport(String metric, LocalDate start, LocalDate end) {
        // ...
    }

    public Report weeklyReport(String metric, LocalDate start, LocalDate end) {
        // ...
    }

    public Report comparison(String metric, LocalDate start, LocalDate end,
                              LocalDate compareStart, LocalDate compareEnd) {
        // ...
    }

    public List<Double> rawData(String metric, LocalDate start, LocalDate end) {
        // ...
    }
}

After — a Period object replaces the clump:

public class Period {
    private final LocalDate start;
    private final LocalDate end;

    public Period(LocalDate start, LocalDate end) {
        if (end.isBefore(start)) {
            throw new IllegalArgumentException("End before start");
        }
        this.start = start;
        this.end = end;
    }

    public long days() {
        return ChronoUnit.DAYS.between(start, end);
    }

    public boolean overlaps(Period other) {
        return !this.end.isBefore(other.start) && !other.end.isBefore(this.start);
    }

    // getters...
}

public class Analytics {
    public Report dailyReport(String metric, Period period) { /* ... */ }
    public Report weeklyReport(String metric, Period period) { /* ... */ }
    public Report comparison(String metric, Period base, Period compare) { /* ... */ }
    public List<Double> rawData(String metric, Period period) { /* ... */ }
}

Validation (end not before start) and behavior (days, overlaps) now live with the data. Every method signature is shorter. And the comparison method no longer has five parameters — it has three, two of which are self-descriptive.

Preserve Whole Object

When you already have an object but are pulling fields out of it just to form the clump, stop destructuring — pass the object.

Before — breaking apart an object to form a clump:

public void scheduleDelivery(Order order) {
    String street = order.getStreet();
    String city = order.getCity();
    String state = order.getState();
    String zip = order.getZipCode();
    double weight = order.getTotalWeight();
    deliveryService.schedule(street, city, state, zip, weight);
}

After — pass the order (or its address):

public void scheduleDelivery(Order order) {
    deliveryService.schedule(order.getAddress(), order.getTotalWeight());
}

The clump disappears because you stop creating it in the first place.
