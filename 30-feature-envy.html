Feature Envy

Description

A method that is more interested in another class's data than in its own. It reaches into another object, pulls out fields, and computes on them — doing work that belongs to the class it's reaching into. The method is in the wrong home. It envies the features of another class.

The telltale sign: a method that makes multiple calls to another object's getters to assemble a result. The logic is about that other object, not about the class the method lives in. If you moved the method to the other class, it would stop needing all those getter calls and would operate directly on the fields.

Feature Envy and Inappropriate Intimacy are related but distinct. Inappropriate Intimacy is about two classes knowing each other's internals. Feature Envy is about a method being in the wrong class — it might use only public getters, but it uses so many of them from one other class that the method clearly belongs there.

Example

public class Customer {
    private String name;
    private String street;
    private String city;
    private String state;
    private String zipCode;
    private String country;

    public String getName() { return name; }
    public String getStreet() { return street; }
    public String getCity() { return city; }
    public String getState() { return state; }
    public String getZipCode() { return zipCode; }
    public String getCountry() { return country; }
}

public class ShippingService {

    public String formatShippingLabel(Customer customer) {
        // six calls to Customer — this method envies Customer's data
        String label = customer.getName() + "\n"
            + customer.getStreet() + "\n"
            + customer.getCity() + ", " + customer.getState()
            + " " + customer.getZipCode() + "\n"
            + customer.getCountry();
        return label;
    }

    public double calculateShippingCost(Customer customer, double weight) {
        // more envy — computing on Customer's data
        if (customer.getCountry().equals("US")) {
            if (customer.getState().equals("HI")
                    || customer.getState().equals("AK")) {
                return weight * 3.50;
            }
            return weight * 1.50;
        }
        return weight * 5.00;
    }
}

formatShippingLabel calls six getters on Customer and uses zero fields from ShippingService. It's a Customer method that accidentally lives in ShippingService. Similarly, calculateShippingCost makes decisions based entirely on Customer's location data.

What principle does it break?

Cohesion and encapsulation. A method should live with the data it operates on. When a method lives in one class but operates primarily on another class's data, cohesion is broken — the method isn't cohesive with its home class. It also weakens the data-owning class: knowledge about how to format or use the data is scattered across the codebase instead of being concentrated where the data lives.

How to fix it

Move Method — move the envious method to the class it envies

public class Customer {
    private String name;
    private String street;
    private String city;
    private String state;
    private String zipCode;
    private String country;

    public String formatShippingLabel() {
        return name + "\n"
            + street + "\n"
            + city + ", " + state + " " + zipCode + "\n"
            + country;
    }

    public boolean isDomestic() {
        return "US".equals(country);
    }

    public boolean isRemoteUS() {
        return isDomestic()
            && ("HI".equals(state) || "AK".equals(state));
    }

    // getters that are still needed for other purposes
    public String getCountry() { return country; }
}
public class ShippingService {

    public String getShippingLabel(Customer customer) {
        return customer.formatShippingLabel();
    }

    public double calculateShippingCost(Customer customer, double weight) {
        if (customer.isRemoteUS()) return weight * 3.50;
        if (customer.isDomestic()) return weight * 1.50;
        return weight * 5.00;
    }
}

formatShippingLabel now lives on Customer, where it has direct access to the fields — no getters needed. ShippingService asks Customer questions about itself (isDomestic, isRemoteUS) instead of interrogating its fields.

Extract Class when the envy targets a subset of fields

If the envied data is a cluster of related fields (like the address fields above), extract them into their own class first, then move the method there.

public class Address {
    private String street;
    private String city;
    private String state;
    private String zipCode;
    private String country;

    public String formatLabel() {
        return street + "\n" + city + ", " + state + " " + zipCode + "\n" + country;
    }

    public boolean isDomestic() { return "US".equals(country); }
    public boolean isRemoteUS() {
        return isDomestic() && ("HI".equals(state) || "AK".equals(state));
    }
}

public class Customer {
    private String name;
    private Address address;

    public String formatShippingLabel() {
        return name + "\n" + address.formatLabel();
    }
}

This combines Extract Class (Data Clumps fix) with Move Method (Feature Envy fix) — two smells eliminated in one refactoring.
