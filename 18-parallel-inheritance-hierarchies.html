Parallel Inheritance Hierarchies

Description

Every time you add a subclass to one hierarchy, you find yourself adding a matching subclass to another hierarchy. Create ExpressShipping and you also need ExpressShippingCalculator. Create PdfReport and you also need PdfReportFormatter. The two hierarchies grow in lockstep — they're parallel lines that must always stay in sync.

This is actually a special case of Shotgun Surgery: adding one conceptual type forces edits in two (or more) places. But it's specific enough and common enough that Fowler calls it out by name. The parallel hierarchies are often connected by a naming convention — one hierarchy uses the prefix or suffix of the other — which is a telltale sign.

Example

// Hierarchy 1: the shapes
public abstract class Shape {
    public abstract double area();
}

public class Circle extends Shape {
    private double radius;
    public Circle(double radius) { this.radius = radius; }
    public double area() { return Math.PI * radius * radius; }
    public double getRadius() { return radius; }
}

public class Rectangle extends Shape {
    private double width, height;
    public Rectangle(double w, double h) { this.width = w; this.height = h; }
    public double area() { return width * height; }
    public double getWidth() { return width; }
    public double getHeight() { return height; }
}

public class Triangle extends Shape {
    private double base, height;
    public Triangle(double b, double h) { this.base = b; this.height = h; }
    public double area() { return 0.5 * base * height; }
    public double getBase() { return base; }
    public double getHeight() { return height; }
}
// Hierarchy 2: the renderers — mirrors the shape hierarchy exactly
public abstract class ShapeRenderer {
    public abstract void render(Shape shape);
}

public class CircleRenderer extends ShapeRenderer {
    @Override
    public void render(Shape shape) {
        Circle c = (Circle) shape;
        System.out.println("Drawing circle with radius " + c.getRadius());
    }
}

public class RectangleRenderer extends ShapeRenderer {
    @Override
    public void render(Shape shape) {
        Rectangle r = (Rectangle) shape;
        System.out.println("Drawing rect " + r.getWidth() + "x" + r.getHeight());
    }
}

public class TriangleRenderer extends ShapeRenderer {
    @Override
    public void render(Shape shape) {
        Triangle t = (Triangle) shape;
        System.out.println("Drawing triangle base=" + t.getBase()
                         + " h=" + t.getHeight());
    }
}

Three shapes, three renderers. Add a Pentagon and you must add a PentagonRenderer. Forget the renderer and the system crashes with a ClassCastException at runtime — there's nothing at compile time that enforces the pairing.

The same pattern appears with serializers, validators, factory methods, and test doubles — any parallel hierarchy that must stay in sync.

What principle does it break?

DRY and cohesion. The knowledge of "how to handle a specific shape" is split across two hierarchies. The concept of a circle lives partly in Circle and partly in CircleRenderer. Neither class tells the whole story, and they must be kept in sync manually.

It also creates fragile coupling: the renderer hierarchy depends on the exact structure of the shape hierarchy, typically through downcasts ((Circle) shape), which are unsafe and break when the hierarchies drift apart.

How to fix it

Collapse the hierarchy — move rendering into the shape

If each renderer only serves one shape, the simplest fix is to move the rendering logic into the shape itself:

public abstract class Shape {
    public abstract double area();
    public abstract void render();
}

public class Circle extends Shape {
    private double radius;

    public Circle(double radius) { this.radius = radius; }

    public double area() { return Math.PI * radius * radius; }

    @Override
    public void render() {
        System.out.println("Drawing circle with radius " + radius);
    }
}

public class Rectangle extends Shape {
    private double width, height;

    public Rectangle(double w, double h) { this.width = w; this.height = h; }

    public double area() { return width * height; }

    @Override
    public void render() {
        System.out.println("Drawing rect " + width + "x" + height);
    }
}

public class Triangle extends Shape {
    private double base, height;

    public Triangle(double b, double h) { this.base = b; this.height = h; }

    public double area() { return 0.5 * base * height; }

    @Override
    public void render() {
        System.out.println("Drawing triangle base=" + base + " h=" + height);
    }
}

The renderer hierarchy disappears entirely. Add a Pentagon and you add one class with one render() method.

Visitor pattern — when collapsing isn't appropriate

Sometimes you can't or shouldn't put rendering inside the shape — perhaps shapes are a domain model that shouldn't know about presentation, or there are multiple rendering strategies (SVG, Canvas, PDF). The Visitor pattern lets you add operations without a parallel hierarchy:

public interface ShapeVisitor<T> {
    T visitCircle(Circle circle);
    T visitRectangle(Rectangle rectangle);
    T visitTriangle(Triangle triangle);
}

public abstract class Shape {
    public abstract double area();
    public abstract <T> T accept(ShapeVisitor<T> visitor);
}

public class Circle extends Shape {
    private double radius;

    public Circle(double radius) { this.radius = radius; }
    public double area() { return Math.PI * radius * radius; }
    public double getRadius() { return radius; }

    @Override
    public <T> T accept(ShapeVisitor<T> visitor) {
        return visitor.visitCircle(this);
    }
}

public class Rectangle extends Shape {
    private double width, height;

    public Rectangle(double w, double h) { this.width = w; this.height = h; }
    public double area() { return width * height; }
    public double getWidth() { return width; }
    public double getHeight() { return height; }

    @Override
    public <T> T accept(ShapeVisitor<T> visitor) {
        return visitor.visitRectangle(this);
    }
}

public class Triangle extends Shape {
    private double base, height;

    public Triangle(double b, double h) { this.base = b; this.height = h; }
    public double area() { return 0.5 * base * height; }
    public double getBase() { return base; }
    public double getHeight() { return height; }

    @Override
    public <T> T accept(ShapeVisitor<T> visitor) {
        return visitor.visitTriangle(this);
    }
}
public class ConsoleRenderer implements ShapeVisitor<Void> {
    public Void visitCircle(Circle c) {
        System.out.println("Drawing circle r=" + c.getRadius());
        return null;
    }

    public Void visitRectangle(Rectangle r) {
        System.out.println("Drawing rect " + r.getWidth() + "x" + r.getHeight());
        return null;
    }

    public Void visitTriangle(Triangle t) {
        System.out.println("Drawing triangle b=" + t.getBase() + " h=" + t.getHeight());
        return null;
    }
}

public class SvgExporter implements ShapeVisitor<String> {
    public String visitCircle(Circle c) {
        return "<circle r=\"" + c.getRadius() + "\"/>";
    }

    public String visitRectangle(Rectangle r) {
        return "<rect width=\"" + r.getWidth() + "\" height=\"" + r.getHeight() + "\"/>";
    }

    public String visitTriangle(Triangle t) {
        return "<polygon points=\"...\"/>";
    }
}

The parallel hierarchy is gone. Adding a new operation (like SvgExporter) means one new class. The trade-off: adding a new shape requires updating every visitor — but the compiler enforces this (you must implement the new visit method), so you can't silently forget it.

Bridge pattern — when both hierarchies need to grow

If both dimensions genuinely need independent extensibility (shapes grow and renderers grow), use the Bridge pattern to decouple them — as covered in the Combinatorial Explosion smell. The key insight is the same: separate the two axes of variation so they don't multiply.
