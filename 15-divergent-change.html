Divergent Change

Description

A single class that needs to be modified for many different, unrelated reasons. You open Employee to change tax logic. Next week you open it again to change the reporting format. The week after that you open it again to update the validation rules. Each change is for a different concern, yet they all converge on the same class. The class is a magnet for change — not because it's central to the design, but because it has absorbed responsibilities that don't belong together.

Divergent Change is the opposite of Shotgun Surgery. In Shotgun Surgery, one change touches many classes. In Divergent Change, many changes touch one class. They're mirror images of the same underlying problem — misplaced responsibility — but they feel different in practice. Divergent Change feels like "this class keeps changing for reasons that have nothing to do with each other."

        if (order.getCustomer().isVip()) return calculateTotal(order) * 0.10;

Five unrelated reasons to change this class: validation rules change, pricing/tax rules change, the database schema changes, the email template changes, the report format changes. Each change is a different concern, but they all require opening and editing OrderProcessor.

What principle does it break?

Single Responsibility Principle (SRP). A class should have only one reason to change. OrderProcessor has at least five. Each responsibility is an axis of change, and they're all independent — tax law has nothing to do with email templates — yet they're coupled in the same file. Changing one risks accidentally breaking another.

How to fix it

The primary refactoring is Extract Class: pull each responsibility into its own class.

public class OrderValidator {
    public boolean validate(Order order) {
        if (order.getItems().isEmpty()) return false;
        if (order.getCustomer() == null) return false;
        for (OrderItem item : order.getItems()) {
            if (item.getQuantity() <= 0) return false;
            if (item.getPrice() < 0) return false;
            if (!inventoryCheck(item)) return false;
        }
        return true;
    }

    private boolean inventoryCheck(OrderItem item) { return true; }
}

public class OrderPricingService {
    public double calculateTotal(Order order) {
        double subtotal = order.getItems().stream()
            .mapToDouble(i -> i.getPrice() * i.getQuantity())
            .sum();
        double discount = getDiscount(order, subtotal);
        double tax = calculateTax(order, subtotal - discount);
        return subtotal - discount + tax;
    }

    private double getDiscount(Order order, double subtotal) {
        if (order.getCustomer().isVip()) return subtotal * 0.10;
        return 0;
    }

    private double calculateTax(Order order, double taxable) {
        return taxable * 0.08;
    }
}

public class OrderRepository {
    public void save(Order order) { /* database logic */ }
    public Order findById(String id) { return null; }
}

public class OrderNotificationService {
    public void notifyCustomer(Order order) {
        String body = formatConfirmation(order);
        // sends email
    }

    private String formatConfirmation(Order order) { return ""; }
}

public class OrderReportGenerator {
    public String generateReport(List<Order> orders) {
        // formats CSV
        return "";
    }
}

Now each class has exactly one reason to change. Tax law changes? Edit OrderPricingService. Database migration? Edit OrderRepository. Email redesign? Edit OrderNotificationService. No class is touched for reasons that aren't its own.

If the original class was being used as a coordinator, you can keep a thin facade:

public class OrderProcessor {
    private final OrderValidator validator;
    private final OrderPricingService pricing;
    private final OrderRepository repository;
    private final OrderNotificationService notifications;

    public OrderProcessor(OrderValidator validator,
                           OrderPricingService pricing,
                           OrderRepository repository,
                           OrderNotificationService notifications) {
        this.validator = validator;
        this.pricing = pricing;
        this.repository = repository;
        this.notifications = notifications;
    }

    public void process(Order order) {
        if (!validator.validate(order)) {
            throw new IllegalArgumentException("Invalid order");
        }
        double total = pricing.calculateTotal(order);
        order.setTotal(total);
        repository.save(order);
        notifications.notifyCustomer(order);
    }
}

The coordinator delegates — it doesn't contain the logic. Each concern lives in its own class, changes independently, and is testable in isolation.
