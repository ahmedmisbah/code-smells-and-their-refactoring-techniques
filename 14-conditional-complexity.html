Conditional Complexity

Also known as: Complex Conditional

Description

Conditional logic that has grown so tangled — deeply nested if/else trees, long chains of && and ||, repeated conditions scattered across methods — that understanding what the code actually decides requires tracing every branch with a pencil. The conditions may have started simple, but edge cases, exceptions, and business rule changes have accumulated until the logic reads like a legal contract.

This smell overlaps with Switch Statements but is broader: it covers any conditional structure that has become hard to read, regardless of whether it's a switch, a chain of if/else, or a thicket of ternary operators. The key indicator is that you read the condition and don't immediately understand when and why each branch executes.

Example

public class InsuranceQuoteCalculator {

    public double calculatePremium(Applicant applicant) {
        double base = 500;

        if (applicant.getAge() < 25) {
            base *= 1.5;
            if (applicant.hasAccidents()) {
                base *= 2.0;
                if (applicant.getAccidentCount() > 3) {
                    base *= 1.5;
                    if (applicant.getLastAccidentYearsAgo() < 1) {
                        base *= 1.3;
                    }
                }
            } else if (applicant.getYearsLicensed() < 2) {
                base *= 1.3;
            }
        } else if (applicant.getAge() < 65) {
            if (applicant.hasAccidents()) {
                base *= 1.3;
                if (applicant.getAccidentCount() > 3) {
                    base *= 1.2;
                }
            }
            if (applicant.getCreditScore() < 600) {
                base *= 1.4;
            } else if (applicant.getCreditScore() > 750) {
                base *= 0.9;
            }
        } else {
            base *= 1.2;
            if (applicant.hasMedicalConditions()) {
                base *= 1.5;
                if (applicant.getMedicalRisk().equals("high")) {
                    base *= 1.3;
                }
            }
        }

        if (applicant.isMultiPolicy()) {
            base *= 0.85;
        }
        if (applicant.isGoodStudent()) {
            base *= 0.9;
        }
        if (applicant.hasDefensiveDrivingCourse()) {
            base *= 0.95;
        }
        if (applicant.getVehicle().getAge() > 10) {
            base *= 0.9;
        } else if (applicant.getVehicle().isLuxury()) {
            base *= 1.4;
        }

        return base;
    }
}

Five levels of nesting in the age-based section. Multipliers scattered across dozens of conditions. The method is a single block of interleaved concerns: age risk, accident history, credit score, medical conditions, discounts, and vehicle factors — all tangled in one tree. Understanding the premium for a 23-year-old with two accidents and a good credit score requires tracing through nested branches manually.

What principle does it break?

Single Responsibility Principle and readability. The method handles age-based risk, accident risk, credit risk, medical risk, and discount eligibility — five distinct policies tangled together. Each is a separate business rule that should be independently readable, testable, and changeable.

It also violates Open/Closed Principle: adding a new risk factor means weaving more conditions into an already unreadable tree.

How to fix it

Decompose Conditional

Extract each branch into a method whose name explains why the branch exists.

public double calculatePremium(Applicant applicant) {
    double base = 500;

    base *= getAgeRiskFactor(applicant);
    base *= getAccidentRiskFactor(applicant);
    base *= getCreditFactor(applicant);
    base *= getMedicalFactor(applicant);
    base *= getDiscountFactor(applicant);
    base *= getVehicleFactor(applicant);

    return base;
}

private double getAgeRiskFactor(Applicant a) {
    if (a.getAge() < 25) return 1.5;
    if (a.getAge() >= 65) return 1.2;
    return 1.0;
}

private double getAccidentRiskFactor(Applicant a) {
    if (!a.hasAccidents()) {
        if (a.getAge() < 25 && a.getYearsLicensed() < 2) return 1.3;
        return 1.0;
    }
    double factor = (a.getAge() < 25) ? 2.0 : 1.3;
    if (a.getAccidentCount() > 3) factor *= 1.5;
    if (a.getAge() < 25 && a.getLastAccidentYearsAgo() < 1) factor *= 1.3;
    return factor;
}

private double getCreditFactor(Applicant a) {
    if (a.getAge() < 25 || a.getAge() >= 65) return 1.0;
    if (a.getCreditScore() < 600) return 1.4;
    if (a.getCreditScore() > 750) return 0.9;
    return 1.0;
}

private double getMedicalFactor(Applicant a) {
    if (a.getAge() < 65 || !a.hasMedicalConditions()) return 1.0;
    if (a.getMedicalRisk().equals("high")) return 1.5 * 1.3;
    return 1.5;
}

private double getDiscountFactor(Applicant a) {
    double factor = 1.0;
    if (a.isMultiPolicy()) factor *= 0.85;
    if (a.isGoodStudent()) factor *= 0.9;
    if (a.hasDefensiveDrivingCourse()) factor *= 0.95;
    return factor;
}

private double getVehicleFactor(Applicant a) {
    if (a.getVehicle().getAge() > 10) return 0.9;
    if (a.getVehicle().isLuxury()) return 1.4;
    return 1.0;
}

The top-level method now reads as a list of named factors. Each factor method is flat, focused, and independently testable.

Replace Conditional with Strategy

When the branching represents fundamentally different algorithms (not just different constants), extract each into a strategy.

Before — different pricing logic by age bracket:

if (age < 25) {
    // young driver pricing — 40 lines
} else if (age < 65) {
    // standard pricing — 30 lines
} else {
    // senior pricing — 35 lines
}

After — a strategy per bracket:

public interface AgePricingStrategy {
    double applyRiskFactor(Applicant applicant, double basePremium);
}

public class YoungDriverPricing implements AgePricingStrategy {
    @Override
    public double applyRiskFactor(Applicant applicant, double base) {
        // young driver specific logic, clean and focused
    }
}

public class StandardPricing implements AgePricingStrategy { /* ... */ }
public class SeniorPricing implements AgePricingStrategy { /* ... */ }

A factory selects the strategy:

AgePricingStrategy strategy = AgePricingStrategyFactory.forAge(applicant.getAge());
double adjusted = strategy.applyRiskFactor(applicant, base);

Each age bracket's logic lives in its own class — testable, replaceable, and invisible to the others.

Replace Nested Conditional with Guard Clauses

When deep nesting comes from checking preconditions, flatten it with early returns.

Before — deeply nested validation:

public double calculateDiscount(Order order) {
    if (order != null) {
        if (order.isPaid()) {
            if (order.getItems().size() > 5) {
                if (order.getTotal() > 100) {
                    return order.getTotal() * 0.10;
                }
            }
        }
    }
    return 0;
}

After — guard clauses flatten the logic:

public double calculateDiscount(Order order) {
    if (order == null) return 0;
    if (!order.isPaid()) return 0;
    if (order.getItems().size() <= 5) return 0;
    if (order.getTotal() <= 100) return 0;

    return order.getTotal() * 0.10;
}

Four levels of nesting become zero. The happy path is at the bottom, clearly visible. Each guard clause is a single, readable check.
