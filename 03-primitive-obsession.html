Primitive Obsession

Description

Using primitive types (strings, integers, doubles, booleans) to represent domain concepts that deserve their own classes. Instead of creating a Money type, you pass around double. Instead of a PhoneNumber class, you use String. Instead of a Temperature object, you use int with a comment saying "in Celsius." The primitives carry no meaning, no validation, and no behavior — they're just raw values drifting through your codebase, relying on every caller to remember what they represent and how to handle them correctly.

This smell often starts innocently. A string for an email address seems fine — until you need to validate it in twelve places. A pair of doubles for latitude and longitude seems fine — until half your methods take six parameters that are all double and you swap two of them by accident.

Example

public class Patient {
    private String name;
    private String bloodType;          // "A+", "O-", "AB+", etc.
    private int    heightCm;
    private double weightKg;
    private String phoneNumber;
    private double temperature;        // Celsius? Fahrenheit? Who knows.
    private String currency;           // "USD", "EUR"
    private double balance;

    public boolean isCompatibleDonor(String recipientBloodType) {
        if (bloodType.equals("O-")) return true;
        if (bloodType.equals("O+") && !recipientBloodType.contains("-")) return true;
        if (bloodType.equals(recipientBloodType)) return true;
        // ... fragile string matching continues
        return false;
    }

    public String formatPhone() {
        // assumes US format — breaks for international numbers
        return "(" + phoneNumber.substring(0, 3) + ") "
             + phoneNumber.substring(3, 6) + "-"
             + phoneNumber.substring(6);
    }

    public boolean hasHighFever() {
        // is this Celsius or Fahrenheit? Caller has to just know.
        return temperature > 38.5;
    }

    public double convertBalanceTo(String targetCurrency, double exchangeRate) {
        // nothing stops you from passing the wrong rate for the wrong currency pair
        return balance * exchangeRate;
    }

    public double calculateBMI() {
        double heightM = heightCm / 100.0;
        return weightKg / (heightM * heightM);
    }
}

Every field is a primitive. Blood type compatibility is scattered string logic. Temperature has no unit. Currency and balance are decoupled — nothing ties "USD" to the balance next to it. Phone formatting assumes a format that isn't enforced. BMI conversion from centimeters to meters is repeated anywhere height is used.

What principle does it break?

Encapsulation. Primitives expose raw data with no protection. A String bloodType accepts "banana" as happily as "O+". Validation, formatting, and comparison logic that belongs with the concept gets scattered across every method that touches it. You end up with the same guard clauses, the same parsing, the same conversion in dozens of places — violating DRY as a direct consequence.

It also erodes type safety. When a method takes (double, double, double), the compiler can't tell you that you swapped latitude and longitude. When it takes (Coordinate location, Temperature temp), it can.

How to fix it

The primary refactoring is Replace Primitive with Object (called Replace Data Value with Object in earlier editions): wrap the primitive in a small class that carries meaning, validation, and behavior.

public class BloodType {
    private final String code;

    private static final Set<String> VALID = Set.of(
        "A+", "A-", "B+", "B-", "AB+", "AB-", "O+", "O-"
    );

    public BloodType(String code) {
        if (!VALID.contains(code)) {
            throw new IllegalArgumentException("Invalid blood type: " + code);
        }
        this.code = code;
    }

    public boolean isUniversalDonor() {
        return code.equals("O-");
    }

    public boolean canDonateTo(BloodType recipient) {
        if (isUniversalDonor()) return true;
        if (code.equals(recipient.code)) return true;
        // ... real compatibility logic lives here, once
        return false;
    }

    @Override
    public String toString() { return code; }

    @Override
    public boolean equals(Object o) {
        return o instanceof BloodType bt && code.equals(bt.code);
    }

    @Override
    public int hashCode() { return code.hashCode(); }
}
public class Temperature {
    private final double value;
    private final Unit unit;

    public enum Unit { CELSIUS, FAHRENHEIT }

    public Temperature(double value, Unit unit) {
        this.value = value;
        this.unit = unit;
    }

    public Temperature toCelsius() {
        if (unit == Unit.CELSIUS) return this;
        return new Temperature((value - 32) * 5.0 / 9.0, Unit.CELSIUS);
    }

    public Temperature toFahrenheit() {
        if (unit == Unit.FAHRENHEIT) return this;
        return new Temperature(value * 9.0 / 5.0 + 32, Unit.FAHRENHEIT);
    }

    public boolean exceeds(Temperature threshold) {
        return this.toCelsius().value > threshold.toCelsius().value;
    }

    public double getValue() { return value; }
    public Unit getUnit() { return unit; }
}
public class Money {
    private final double amount;
    private final Currency currency;

    public Money(double amount, Currency currency) {
        this.amount = amount;
        this.currency = currency;
    }

    public Money convertTo(Currency target, ExchangeRateProvider rates) {
        double rate = rates.getRate(this.currency, target);
        return new Money(amount * rate, target);
    }

    public Money add(Money other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException(
                "Cannot add " + currency + " to " + other.currency
            );
        }
        return new Money(this.amount + other.amount, this.currency);
    }

    // ... subtract, compareTo, etc.
}
public class PhoneNumber {
    private final String digits;
    private final String countryCode;

    public PhoneNumber(String countryCode, String digits) {
        if (!digits.matches("\\d{7,15}")) {
            throw new IllegalArgumentException("Invalid phone digits: " + digits);
        }
        this.countryCode = countryCode;
        this.digits = digits;
    }

    public String format() {
        if (countryCode.equals("1") && digits.length() == 10) {
            return "(" + digits.substring(0, 3) + ") "
                 + digits.substring(3, 6) + "-"
                 + digits.substring(6);
        }
        return "+" + countryCode + " " + digits;
    }
}

Now the Patient class becomes clean:

public class Patient {
    private String name;
    private BloodType bloodType;
    private Height height;
    private Weight weight;
    private PhoneNumber phone;
    private Temperature temperature;
    private Money balance;

    public boolean isCompatibleDonor(Patient recipient) {
        return bloodType.canDonateTo(recipient.bloodType);
    }

    public boolean hasHighFever() {
        return temperature.exceeds(new Temperature(38.5, Temperature.Unit.CELSIUS));
    }

    public double calculateBMI() {
        return weight.toKg() / Math.pow(height.toMeters(), 2);
    }
}

Each concept validates itself, converts itself, and compares itself. The caller can't misuse it — you can't accidentally pass a Temperature where a Money is expected.

Supporting refactorings

Replace Type Code with Subclasses

When a primitive (usually an int or String) is used to distinguish behavior — acting as a type tag — replace it with a class hierarchy.

Before — a string type code drives conditional logic:

public class Employee {
    private String type; // "engineer", "manager", "intern"

    public double getBonus(double salary) {
        if (type.equals("engineer")) {
            return salary * 0.10;
        } else if (type.equals("manager")) {
            return salary * 0.20;
        } else if (type.equals("intern")) {
            return 0;
        }
        throw new IllegalStateException("Unknown type: " + type);
    }
}

After — the type code becomes a hierarchy, and each subclass owns its behavior:

public abstract class Employee {
    public abstract double getBonus(double salary);
}

public class Engineer extends Employee {
    @Override
    public double getBonus(double salary) {
        return salary * 0.10;
    }
}

public class Manager extends Employee {
    @Override
    public double getBonus(double salary) {
        return salary * 0.20;
    }
}

public class Intern extends Employee {
    @Override
    public double getBonus(double salary) {
        return 0;
    }
}

Replace Type Code with State/Strategy

When the type can change at runtime (an employee gets promoted), subclassing the host object doesn't work — you can't change an object's class. Instead, extract the varying behavior into a separate strategy or state object.

Before — a mutable type code with conditional logic:

public class Subscription {
    private String tier; // "free", "basic", "premium"

    public void upgrade() {
        if (tier.equals("free")) tier = "basic";
        else if (tier.equals("basic")) tier = "premium";
    }

    public int getMaxDevices() {
        if (tier.equals("free")) return 1;
        if (tier.equals("basic")) return 3;
        return 10;
    }

    public boolean canDownloadOffline() {
        return tier.equals("premium");
    }
}

After — behavior lives in swappable strategy objects:

public interface SubscriptionTier {
    SubscriptionTier upgrade();
    int getMaxDevices();
    boolean canDownloadOffline();
}

public class FreeTier implements SubscriptionTier {
    public SubscriptionTier upgrade() { return new BasicTier(); }
    public int getMaxDevices() { return 1; }
    public boolean canDownloadOffline() { return false; }
}

public class BasicTier implements SubscriptionTier {
    public SubscriptionTier upgrade() { return new PremiumTier(); }
    public int getMaxDevices() { return 3; }
    public boolean canDownloadOffline() { return false; }
}

public class PremiumTier implements SubscriptionTier {
    public SubscriptionTier upgrade() { return this; }
    public int getMaxDevices() { return 10; }
    public boolean canDownloadOffline() { return true; }
}

public class Subscription {
    private SubscriptionTier tier;

    public Subscription() {
        this.tier = new FreeTier();
    }

    public void upgrade() {
        this.tier = tier.upgrade();
    }

    public int getMaxDevices() {
        return tier.getMaxDevices();
    }

    public boolean canDownloadOffline() {
        return tier.canDownloadOffline();
    }
}

The Subscription can change tier at runtime without conditionals, and adding a new tier means adding a class — not editing every if/else chain.

Introduce Parameter Object

When multiple related primitives travel together through method signatures, bundle them into a single object. (We covered this in detail under Long Method; the same technique applies here — the motivation is just different. In Long Method the goal is to shrink parameter lists; here the goal is to give the concept a name and a home for behavior.)
