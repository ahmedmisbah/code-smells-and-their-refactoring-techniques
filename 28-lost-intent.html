Lost Intent

Description

Code that doesn't easily communicate its author's intention. The purpose is obscured by complexity, unintuitive names, poor organization, blended algorithm steps, or reliance on side effects. You read the code and can't answer the basic question: "what is this trying to do?"

Lost Intent is a meta-smell — it's the cumulative effect of many other smells working together. Mysterious names hide what things are. Long methods blend steps together. Conditional complexity buries the decision logic. Side effects create invisible behavior. Comments try to compensate but rot over time. The result is code that is hazardous: hard to understand, hard to extend, and hard to modify.

Industrial Logic contrasts intention-revealing code with lost intent code across four dimensions:

Intention-Revealing CodeLost Intent
The purpose of the code is clear.Low-level or complex logic obscures the code's purpose.
You can easily find the code you seek.The location of code makes little sense.
Every step of an algorithm is clearly defined.The steps of the algorithm are blended together.
Code depends on no side effects.Code depends on side effects of operations/functions.

Example

A Loan class with five overloaded constructors — each creating a different kind of loan, but none revealing which kind:

public class Loan {

    public Loan(double commitment, int riskRating, int maturity) {
        // creates... what kind of loan?
    }

    public Loan(double commitment, int riskRating, int maturity, Date expiry) {
        // is this a term loan? a revolver? an RCTL?
    }

    public Loan(double commitment, double outstanding, int riskRating,
                int maturity, Date expiry) {
        // more parameters, still no clue
    }

    public Loan(CapitalStrategy capitalStrategy, double commitment,
                int riskRating, int maturity, Date expiry) {
        // the strategy might hint, but the constructor doesn't say
    }

    public Loan(CapitalStrategy capitalStrategy, double commitment,
                double outstanding, int riskRating, int maturity, Date expiry) {
        // the "full" constructor — but full of what?
    }
}

A developer looking at this class has no idea which constructor to call to create a Term Loan versus a Revolver versus an RCTL. The intent — what kind of loan each constructor produces — is lost inside parameter combinations that only the original author understood. You have to read the implementation of each constructor, compare the parameter lists, and reverse-engineer the mapping.

Here's another example — a method whose steps are blended together:

public double process(List<Map<String, Object>> records, String mode, boolean flag) {
    double r = 0;
    List<Map<String, Object>> tmp = new ArrayList<>();
    for (Map<String, Object> rec : records) {
        if (flag && rec.get("status") != null
                && ((String) rec.get("status")).startsWith("A")
                && !mode.equals("X")) {
            double v = ((Number) rec.get("val")).doubleValue();
            if (mode.equals("M")) {
                v *= 1.1;
            } else if (mode.equals("D")) {
                v *= 0.9;
            }
            r += v;
            rec.put("processed", true);  // side effect
            tmp.add(rec);
        }
    }
    records.clear();       // side effect: clears the input
    records.addAll(tmp);   // side effect: replaces input contents
    return r;
}

What does this method do? The name process says nothing. The parameters mode and flag are opaque. The filtering condition is a wall of casts and string comparisons. The method computes a total and mutates the input list and marks records as processed — three blended purposes. The intent is completely buried.

What principle does it break?

Intention-Revealing Design. Code should communicate what it does through its names, structure, and organization — not force the reader to reverse-engineer meaning from implementation details. When intent is lost, every reader pays a tax: they must reconstruct the author's thinking from cryptic artifacts, which is slow, error-prone, and demoralizing.

Lost Intent is also a compounding violation of many principles at once: SRP (blended steps), least surprise (side effects), readability (poor names), and separation of concerns (algorithm steps tangled together). It's the smell you get when multiple individual smells are left untreated until the code's purpose is no longer recoverable from a casual reading.

How to fix it

Replace Constructors with Creation Methods

For the Loan example, replace the ambiguous constructors with named static factory methods that reveal what kind of loan they create.

Before — which constructor creates a Term Loan?

Loan termLoan = new Loan(commitment, riskRating, maturity);
// or is it this one?
Loan termLoan = new Loan(capitalStrategy, commitment, riskRating, maturity, expiry);

After — creation methods make the intent explicit:

public class Loan {
    private final CapitalStrategy capitalStrategy;
    private final double commitment;
    private final double outstanding;
    private final int riskRating;
    private final int maturity;
    private final Date expiry;

    private Loan(CapitalStrategy capitalStrategy, double commitment,
                  double outstanding, int riskRating,
                  int maturity, Date expiry) {
        this.capitalStrategy = capitalStrategy;
        this.commitment = commitment;
        this.outstanding = outstanding;
        this.riskRating = riskRating;
        this.maturity = maturity;
        this.expiry = expiry;
    }

    public static Loan createTermLoan(double commitment, int riskRating,
                                       int maturity) {
        return new Loan(null, commitment, 0, riskRating, maturity, null);
    }

    public static Loan createTermLoan(CapitalStrategy strategy,
                                       double commitment, int riskRating,
                                       int maturity) {
        return new Loan(strategy, commitment, 0, riskRating, maturity, null);
    }

    public static Loan createRevolver(double commitment, double outstanding,
                                       int riskRating, Date expiry) {
        return new Loan(null, commitment, outstanding, riskRating, 0, expiry);
    }

    public static Loan createRevolver(CapitalStrategy strategy,
                                       double commitment, double outstanding,
                                       int riskRating, Date expiry) {
        return new Loan(strategy, commitment, outstanding, riskRating, 0, expiry);
    }

    public static Loan createRCTL(double commitment, double outstanding,
                                    int riskRating, int maturity, Date expiry) {
        return new Loan(null, commitment, outstanding, riskRating, maturity, expiry);
    }

    public static Loan createRCTL(CapitalStrategy strategy, double commitment,
                                    double outstanding, int riskRating,
                                    int maturity, Date expiry) {
        return new Loan(strategy, commitment, outstanding, riskRating,
                        maturity, expiry);
    }
}

Now the calling code reveals its intent:

Loan term = Loan.createTermLoan(500_000, 3, 36);
Loan revolver = Loan.createRevolver(250_000, 50_000, 2, expiryDate);
Loan rctl = Loan.createRCTL(strategy, 1_000_000, 200_000, 4, 60, expiryDate);

Each line says exactly what it's creating. The constructor is private — you must go through a named creation method.

Extract Method + Rename — reveal the algorithm's steps

For the blended-steps example, separate each concern and name it.

After:

public class RecordProcessor {

    public ProcessingResult process(List<Record> records) {
        List<Record> eligible = filterEligible(records);
        List<Record> adjusted = applyAdjustments(eligible);
        double total = computeTotal(adjusted);
        return new ProcessingResult(adjusted, total);
    }

    private List<Record> filterEligible(List<Record> records) {
        return records.stream()
            .filter(Record::isActive)
            .collect(Collectors.toList());
    }

    private List<Record> applyAdjustments(List<Record> records) {
        return records.stream()
            .map(this::applyAdjustment)
            .collect(Collectors.toList());
    }

    private Record applyAdjustment(Record record) {
        if (record.getMode() == Mode.MARKUP) {
            return record.withValue(record.getValue() * 1.1);
        }
        if (record.getMode() == Mode.DISCOUNT) {
            return record.withValue(record.getValue() * 0.9);
        }
        return record;
    }

    private double computeTotal(List<Record> records) {
        return records.stream()
            .mapToDouble(Record::getValue)
            .sum();
    }
}

The method process now reads as a clear sequence of named steps: filter, adjust, total. No side effects — the input isn't mutated. No cryptic flags — Mode.MARKUP replaces "M". No type casting — Record replaces Map<String, Object>. The intent is recovered.

General strategies for recovering lost intent

Lost Intent is fixed by treating the individual smells that caused it:

Mysterious names → Rename Method, Rename Variable, Rename Class
Blended steps → Extract Method (one method per step)
Side effects → Separate Commands from Queries (see Side Effect smell)
Complex conditionals → Decompose Conditional, Replace Conditional with Polymorphism
Ambiguous constructors → Replace Constructor with Creation Method (Factory Method)
Data hidden in generic structures (Map<String, Object>) → Replace with typed domain objects

The common thread: make every name, every method boundary, and every class boundary serve as documentation of intent. When the code's purpose is visible in its structure, comments become optional and the intent is never lost.
