Indecent Exposure

Description

Classes, methods, or fields that are more visible than they need to be — public when they should be package-private, protected when they should be private, or exposed as API when they're implementation details. The code exposes its internals to every class in the system when only a handful of classes (or none) actually need access.

Every public element is a promise: "you can depend on this, and I won't break it." Indecent exposure makes promises the class never intended to keep. Other classes start depending on internal helpers, factory methods meant for one specific caller, or data structures that were supposed to be hidden. Now you can't refactor without breaking dependents you never knew existed.

Kerievsky identifies this as a particular concern with public constructors — making a constructor public when creation should be controlled through a factory or builder exposes construction details that should be hidden.

Example

public class ReportEngine {

    // should be private — only used internally
    public List<String> rawData;

    // should be private — internal helper
    public String sanitize(String input) {
        return input.replaceAll("[^a-zA-Z0-9 ]", "");
    }

    // should be private — intermediate step
    public List<String[]> parseRows(List<String> lines) {
        return lines.stream()
            .map(line -> line.split(","))
            .collect(Collectors.toList());
    }

    // should be private — formatting detail
    public String formatCell(String value, int width) {
        return String.format("%-" + width + "s", value);
    }

    // the actual public API
    public String generateReport(List<String> data) {
        this.rawData = data;
        List<String> cleaned = data.stream()
            .map(this::sanitize)
            .collect(Collectors.toList());
        List<String[]> rows = parseRows(cleaned);
        StringBuilder sb = new StringBuilder();
        for (String[] row : rows) {
            for (String cell : row) {
                sb.append(formatCell(cell, 20));
            }
            sb.append("\n");
        }
        return sb.toString();
    }
}

sanitize, parseRows, formatCell, and rawData are all public, but they're implementation details of generateReport. Other classes start calling sanitize for their own purposes. Someone reads rawData directly. Now ReportEngine can't change its parsing strategy, rename its helpers, or remove rawData without a system-wide impact study.

What principle does it break?

Information Hiding / Minimum Visibility. Every element should have the narrowest visibility that allows it to do its job. Public elements are part of the contract; private elements are free to change. Exposing more than necessary turns implementation details into accidental contracts.

It also violates encapsulation in the classical sense: the object's internal state and helper methods are visible and accessible to anyone.

How to fix it

Restrict visibility — make things as private as possible

public class ReportEngine {

    // only the public API remains public

    public String generateReport(List<String> data) {
        List<String> cleaned = data.stream()
            .map(this::sanitize)
            .collect(Collectors.toList());
        List<String[]> rows = parseRows(cleaned);
        StringBuilder sb = new StringBuilder();
        for (String[] row : rows) {
            for (String cell : row) {
                sb.append(formatCell(cell, 20));
            }
            sb.append("\n");
        }
        return sb.toString();
    }

    private String sanitize(String input) {
        return input.replaceAll("[^a-zA-Z0-9 ]", "");
    }

    private List<String[]> parseRows(List<String> lines) {
        return lines.stream()
            .map(line -> line.split(","))
            .collect(Collectors.toList());
    }

    private String formatCell(String value, int width) {
        return String.format("%-" + width + "s", value);
    }
}

rawData is gone — it was only needed as a temporary variable, so it becomes a local in generateReport (or disappears). The three helper methods are private. The class now has one public method — that's its contract. Everything else can change freely.

Encapsulate Construction — hide constructors behind factory methods

When construction should be controlled:

Before — public constructor exposes creation details:

public class DatabaseConnection {
    public DatabaseConnection(String url, String user, String password,
                               int poolSize, int timeout, boolean ssl) {
        // ...
    }
}

After — private constructor, public factory:

public class DatabaseConnection {
    private DatabaseConnection(String url, String user, String password,
                                int poolSize, int timeout, boolean ssl) {
        // ...
    }

    public static DatabaseConnection createDefault() {
        return new DatabaseConnection(
            Config.getDbUrl(), Config.getDbUser(), Config.getDbPassword(),
            10, 5000, true);
    }

    public static DatabaseConnection createForTesting() {
        return new DatabaseConnection("jdbc:h2:mem:test", "sa", "", 2, 1000, false);
    }
}

Callers can't construct a connection with arbitrary parameters. The factory methods control what's possible, and the constructor's parameter list is hidden.
