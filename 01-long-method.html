Long Method

Also known as: Long Function (Fowler, 2nd ed.)

Description

A method that contains too many lines of code, typically doing more than one thing. There's no universal threshold, but most authors agree that once a method exceeds roughly 5–10 lines (debatable), it deserves scrutiny. Long methods tend to grow organically — it always feels easier to add a few lines to an existing method than to create a new one — until the method becomes a tangled, hard-to-follow narrative.

Example

public String generateInvoice(Order order) {
    // Validate order
    if (order == null) throw new IllegalArgumentException("Order cannot be null");
    if (order.getItems().isEmpty()) throw new IllegalArgumentException("Order has no items");
    for (OrderItem item : order.getItems()) {
        if (item.getQuantity() <= 0) throw new IllegalArgumentException("Invalid quantity");
        if (item.getPrice() < 0) throw new IllegalArgumentException("Invalid price");
    }

    // Calculate subtotal
    double subtotal = 0;
    for (OrderItem item : order.getItems()) {
        double lineTotal = item.getPrice() * item.getQuantity();
        if (item.getCategory().equals("electronics")) {
            lineTotal *= 0.95; // 5% discount on electronics
        } else if (item.getCategory().equals("books")) {
            lineTotal *= 0.90; // 10% discount on books
        }
        subtotal += lineTotal;
    }

    // Calculate tax
    double tax;
    if (order.getState().equals("CA")) {
        tax = subtotal * 0.0975;
    } else if (order.getState().equals("NY")) {
        tax = subtotal * 0.08875;
    } else if (order.getState().equals("TX")) {
        tax = subtotal * 0.0625;
    } else {
        tax = subtotal * 0.07;
    }

    // Calculate shipping
    double shipping;
    double totalWeight = 0;
    for (OrderItem item : order.getItems()) {
        totalWeight += item.getWeight() * item.getQuantity();
    }
    if (totalWeight < 5) {
        shipping = 5.99;
    } else if (totalWeight < 20) {
        shipping = 12.99;
    } else {
        shipping = 24.99;
    }
    if (subtotal > 100) {
        shipping = 0; // Free shipping over $100
    }

    // Format invoice
    StringBuilder sb = new StringBuilder();
    sb.append("INVOICE\n");
    sb.append("Customer: ").append(order.getCustomerName()).append("\n");
    sb.append("----------------------------\n");
    for (OrderItem item : order.getItems()) {
        sb.append(item.getName())
          .append(" x").append(item.getQuantity())
          .append(" @ $").append(String.format("%.2f", item.getPrice()))
          .append("\n");
    }
    sb.append("----------------------------\n");
    sb.append("Subtotal: $").append(String.format("%.2f", subtotal)).append("\n");
    sb.append("Tax:      $").append(String.format("%.2f", tax)).append("\n");
    sb.append("Shipping: $").append(String.format("%.2f", shipping)).append("\n");
    sb.append("TOTAL:    $").append(String.format("%.2f", subtotal + tax + shipping)).append("\n");
    return sb.toString();
}

What principle does it break?

Single Responsibility Principle (SRP). The method above validates, calculates discounts, computes tax, determines shipping, and formats output — five distinct responsibilities in one place. This also violates the closely related idea of separation of concerns: each block operates at a different conceptual level, yet they're all stitched together sequentially. The result is code that is hard to read, hard to test in isolation, and hard to change without risking unrelated breakage. If the tax rules change, you're editing the same method that handles formatting. If you need the subtotal elsewhere, you can't reuse it — you have to duplicate the logic.

How to fix it

The primary refactoring is Extract Method (called Extract Function in Fowler's 2nd edition): identify each coherent chunk and pull it into its own well-named method.

public String generateInvoice(Order order) {
    validateOrder(order);
    double subtotal = calculateSubtotal(order);
    double tax = calculateTax(order, subtotal);
    double shipping = calculateShipping(order, subtotal);
    return formatInvoice(order, subtotal, tax, shipping);
}

private void validateOrder(Order order) {
    if (order == null) throw new IllegalArgumentException("Order cannot be null");
    if (order.getItems().isEmpty()) throw new IllegalArgumentException("Order has no items");
    for (OrderItem item : order.getItems()) {
        if (item.getQuantity() <= 0) throw new IllegalArgumentException("Invalid quantity");
        if (item.getPrice() < 0) throw new IllegalArgumentException("Invalid price");
    }
}

private double calculateSubtotal(Order order) {
    double subtotal = 0;
    for (OrderItem item : order.getItems()) {
        subtotal += getDiscountedPrice(item) * item.getQuantity();
    }
    return subtotal;
}

private double getDiscountedPrice(OrderItem item) {
    if (item.getCategory().equals("electronics")) return item.getPrice() * 0.95;
    if (item.getCategory().equals("books")) return item.getPrice() * 0.90;
    return item.getPrice();
}

private double calculateTax(Order order, double subtotal) {
    double rate = getTaxRate(order.getState());
    return subtotal * rate;
}

private double getTaxRate(String state) {
    switch (state) {
        case "CA": return 0.0975;
        case "NY": return 0.08875;
        case "TX": return 0.0625;
        default:   return 0.07;
    }
}

private double calculateShipping(Order order, double subtotal) {
    if (subtotal > 100) return 0;
    double totalWeight = order.getItems().stream()
        .mapToDouble(i -> i.getWeight() * i.getQuantity()).sum();
    if (totalWeight < 5)  return 5.99;
    if (totalWeight < 20) return 12.99;
    return 24.99;
}

private String formatInvoice(Order order, double subtotal,
                              double tax, double shipping) {
    StringBuilder sb = new StringBuilder();
    sb.append("INVOICE\n");
    sb.append("Customer: ").append(order.getCustomerName()).append("\n");
    sb.append("----------------------------\n");
    for (OrderItem item : order.getItems()) {
        sb.append(item.getName())
          .append(" x").append(item.getQuantity())
          .append(" @ $").append(String.format("%.2f", item.getPrice()))
          .append("\n");
    }
    sb.append("----------------------------\n");
    sb.append("Subtotal: $").append(String.format("%.2f", subtotal)).append("\n");
    sb.append("Tax:      $").append(String.format("%.2f", tax)).append("\n");
    sb.append("Shipping: $").append(String.format("%.2f", shipping)).append("\n");
    sb.append("TOTAL:    $")
       .append(String.format("%.2f", subtotal + tax + shipping)).append("\n");
    return sb.toString();
}

Now generateInvoice reads like a high-level summary — you can understand what it does without wading through the details. Each extracted method can be tested, reused, and modified independently.

Supporting refactorings

When Extract Method alone isn't enough, these refactorings clear the way.

Replace Temp with Query

Local variables pin logic inside a method because extracted methods can't see them. The fix: replace the variable with a method call.

Before — basePrice blocks extraction of the discount logic:

public double getPrice(Order order) {
    double basePrice = order.getQuantity() * order.getItemPrice();
    double discount;
    if (basePrice > 1000) {
        discount = basePrice * 0.10;
    } else {
        discount = basePrice * 0.05;
    }
    return basePrice - discount;
}

After — basePrice() is now a query, so the discount logic extracts cleanly:

public double getPrice(Order order) {
    return basePrice(order) - discount(order);
}

private double basePrice(Order order) {
    return order.getQuantity() * order.getItemPrice();
}

private double discount(Order order) {
    if (basePrice(order) > 1000) {
        return basePrice(order) * 0.10;
    }
    return basePrice(order) * 0.05;
}

Introduce Parameter Object

When several parameters always travel together, they're begging to become an object. This shrinks parameter lists and gives the concept a name.

Before — date range parameters clutter every method signature:

public List<Event> getEvents(Date start, Date end, String timezone) {
    // ...
}

public double getRevenue(Date start, Date end, String timezone) {
    // ...
}

public List<User> getActiveUsers(Date start, Date end, String timezone) {
    // ...
}

After — a DateRange object replaces the clump:

public class DateRange {
    private final Date start;
    private final Date end;
    private final String timezone;

    public DateRange(Date start, Date end, String timezone) {
        this.start = start;
        this.end = end;
        this.timezone = timezone;
    }

    public boolean contains(Date date) {
        return !date.before(start) && !date.after(end);
    }

    // getters...
}

public List<Event> getEvents(DateRange range) { /* ... */ }
public double getRevenue(DateRange range) { /* ... */ }
public List<User> getActiveUsers(DateRange range) { /* ... */ }

Notice how behavior naturally migrates into the new class (contains). This is a common and valuable side-effect: what starts as a data carrier often grows into a proper object.

Preserve Whole Object

Instead of pulling several values out of an object just to pass them individually, pass the object itself.

Before — three values are extracted only to be passed along:

public void recordWeather(WeatherStation station) {
    double temp = station.getTemperature();
    double humidity = station.getHumidity();
    double windSpeed = station.getWindSpeed();
    logReading(temp, humidity, windSpeed);
}

private void logReading(double temp, double humidity, double windSpeed) {
    if (temp > 100 && humidity < 20 && windSpeed > 30) {
        triggerFireWarning();
    }
    // ...
}

After — pass the whole object:

public void recordWeather(WeatherStation station) {
    logReading(station);
}

private void logReading(WeatherStation station) {
    if (station.getTemperature() > 100
            && station.getHumidity() < 20
            && station.getWindSpeed() > 30) {
        triggerFireWarning();
    }
    // ...
}

The parameter list shrinks and the code is resilient to change — if the warning logic later needs barometric pressure, you don't have to thread a new parameter through every caller.

Replace Method with Method Object

When a method is so loaded with local variables and intertwined logic that extraction feels impossible, promote the entire method into its own class. The locals become fields, and you're free to decompose at will.

Before — a pricing method with deeply entangled locals:

public class Order {
    public double calculateTotal(List<OrderItem> items, Customer customer) {
        double baseTotal = 0;
        double volumeDiscount = 0;
        double loyaltyCredit = 0;
        double taxableAmount = 0;
        double shippingSurcharge = 0;

        for (OrderItem item : items) {
            double linePrice = item.getPrice() * item.getQuantity();
            if (item.getQuantity() > 100) {
                volumeDiscount += linePrice * 0.15;
            } else if (item.getQuantity() > 50) {
                volumeDiscount += linePrice * 0.10;
            }
            if (item.isTaxable()) {
                taxableAmount += linePrice;
            }
            if (item.isOversized()) {
                shippingSurcharge += item.getWeight() * 0.50;
            }
            baseTotal += linePrice;
        }

        if (customer.getLoyaltyYears() > 5) {
            loyaltyCredit = baseTotal * 0.05;
        } else if (customer.getLoyaltyYears() > 2) {
            loyaltyCredit = baseTotal * 0.02;
        }

        double tax = taxableAmount * customer.getTaxRate();
        return baseTotal - volumeDiscount - loyaltyCredit + tax + shippingSurcharge;
    }
}

After — the method becomes a class, locals become fields, and decomposition is straightforward:

public class Order {
    public double calculateTotal(List<OrderItem> items, Customer customer) {
        return new OrderTotalCalculator(items, customer).compute();
    }
}

public class OrderTotalCalculator {
    private final List<OrderItem> items;
    private final Customer customer;
    private double baseTotal;
    private double volumeDiscount;
    private double loyaltyCredit;
    private double taxableAmount;
    private double shippingSurcharge;

    public OrderTotalCalculator(List<OrderItem> items, Customer customer) {
        this.items = items;
        this.customer = customer;
    }

    public double compute() {
        calculateLineItems();
        calculateLoyaltyCredit();
        double tax = calculateTax();
        return baseTotal - volumeDiscount - loyaltyCredit + tax + shippingSurcharge;
    }

    private void calculateLineItems() {
        for (OrderItem item : items) {
            double linePrice = item.getPrice() * item.getQuantity();
            baseTotal += linePrice;
            applyVolumeDiscount(item, linePrice);
            trackTaxable(item, linePrice);
            trackShipping(item);
        }
    }

    private void applyVolumeDiscount(OrderItem item, double linePrice) {
        if (item.getQuantity() > 100) {
            volumeDiscount += linePrice * 0.15;
        } else if (item.getQuantity() > 50) {
            volumeDiscount += linePrice * 0.10;
        }
    }

    private void trackTaxable(OrderItem item, double linePrice) {
        if (item.isTaxable()) {
            taxableAmount += linePrice;
        }
    }

    private void trackShipping(OrderItem item) {
        if (item.isOversized()) {
            shippingSurcharge += item.getWeight() * 0.50;
        }
    }

    private void calculateLoyaltyCredit() {
        if (customer.getLoyaltyYears() > 5) {
            loyaltyCredit = baseTotal * 0.05;
        } else if (customer.getLoyaltyYears() > 2) {
            loyaltyCredit = baseTotal * 0.02;
        }
    }

    private double calculateTax() {
        return taxableAmount * customer.getTaxRate();
    }
}

The locals that made extraction impossible are now fields shared across small, focused methods. Each piece can be understood, tested, and changed independently.

If the long method's core problem is a large switch or cascading if/else dispatching different behaviors, a Strategy pattern may be warranted — but that overlaps with the Switch Statements smell, which we'll cover separately.
