Lazy Class

Also known as: Freeloader

Description

A class that doesn't do enough to justify its existence. It might have been created with good intentions — anticipating future complexity — but that complexity never arrived. Or it might be the residue of a refactoring that moved most of the class's logic elsewhere, leaving behind a thin shell with one trivial method.

Every class has a cost: it adds a name to the vocabulary, a file to navigate, a concept to understand. If the class doesn't carry enough weight to earn that cost, it's overhead. The code would be simpler if the class were inlined into its caller or merged with a neighbor.

Example

public class StringUtils {
    public static String capitalize(String input) {
        if (input == null || input.isEmpty()) return input;
        return input.substring(0, 1).toUpperCase() + input.substring(1);
    }
}

A class with a single trivial method that wraps a one-liner. The caller could do this inline, or the method could live in whatever class actually uses it.

Another common form:

public class OrderWrapper {
    private final Order order;

    public OrderWrapper(Order order) {
        this.order = order;
    }

    public Order getOrder() {
        return order;
    }
}

A class that holds an object and returns it. It delegates everything, adds nothing, and exists only because someone thought a wrapper might be needed someday.

What principle does it break?

Simplicity / YAGNI (You Aren't Gonna Need It). Every class should earn its keep. A class that doesn't carry meaningful behavior or encapsulate meaningful state is conceptual clutter — it adds to the vocabulary without adding to the value. The codebase would be simpler without it.

How to fix it

Inline Class

Merge the lazy class into whatever class uses it.

Before:

public class PhoneNumber {
    private final String number;

    public PhoneNumber(String number) {
        this.number = number;
    }

    public String getNumber() {
        return number;
    }
}

public class Customer {
    private String name;
    private PhoneNumber phone;

    public String getPhone() {
        return phone.getNumber();
    }
}

If PhoneNumber does nothing but hold a string — no validation, no formatting, no behavior — inline it:

After:

public class Customer {
    private String name;
    private String phone;

    public String getPhone() {
        return phone;
    }
}

If the class later grows behavior (formatting, validation), it can be extracted again at that point.

Collapse Hierarchy

When a subclass does nothing beyond what the parent does, merge it into the parent.

Before:

public class Employee {
    private String name;
    private double salary;
    // ... full behavior
}

public class RegularEmployee extends Employee {
    // completely empty — adds nothing
}

After:

public class Employee {
    private String name;
    private double salary;
    // ... full behavior
}
// RegularEmployee is gone — just use Employee directly

When to keep a Lazy Class

A seemingly lazy class is justified when it carries semantic meaning even without much code. A Money class that wraps a double and a String might look lazy, but it prevents mixing currencies and catches type errors at compile time. The behavior is small, but the meaning is large. Judge by value, not by line count.
