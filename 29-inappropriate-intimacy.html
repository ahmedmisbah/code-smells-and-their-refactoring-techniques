Inappropriate Intimacy

Description

Two classes that know too much about each other's internal details — they reach into each other's private fields, call each other's internal methods, or depend on implementation details that should be hidden. The classes are tangled in a relationship where neither can change without breaking the other.

A healthy relationship between classes is through well-defined public interfaces — narrow, stable contracts. Inappropriate intimacy is when classes bypass those contracts and depend on each other's guts. One class reads another's private fields through reflection, or relies on the order of its internal list, or calls a method that was meant to be a private helper. The coupling is deep, fragile, and invisible from the public API.

This smell is especially common between classes that were split from a single class — the split happened, but the two halves still behave as if they're one unit, constantly reaching across the boundary.

Example

public class Order {
    List<OrderItem> items = new ArrayList<>();  // package-private, not truly private
    double discountPercentage;
    Customer customer;

    public void addItem(OrderItem item) {
        items.add(item);
    }
}

public class OrderPrinter {

    public String print(Order order) {
        StringBuilder sb = new StringBuilder();
        sb.append("Customer: ").append(order.customer.name).append("\n");

        // reaches directly into Order's internal list
        for (int i = 0; i < order.items.size(); i++) {
            OrderItem item = order.items.get(i);
            double lineTotal = item.price * item.quantity;

            // knows about Order's internal discount field
            if (order.discountPercentage > 0) {
                lineTotal *= (1 - order.discountPercentage / 100);
            }
            sb.append(item.name).append(": $")
              .append(String.format("%.2f", lineTotal)).append("\n");
        }

        // computes the total — but this is Order's job
        double total = 0;
        for (OrderItem item : order.items) {
            total += item.price * item.quantity;
        }
        total *= (1 - order.discountPercentage / 100);
        sb.append("Total: $").append(String.format("%.2f", total));
        return sb.toString();
    }
}

public class OrderValidator {

    public boolean validate(Order order) {
        // reaches into Order's internals again
        if (order.items.isEmpty()) return false;
        if (order.customer == null) return false;

        // knows about the internal structure of items
        for (OrderItem item : order.items) {
            if (item.price < 0) return false;
            if (item.quantity <= 0) return false;
        }

        // knows about discount rules that belong to Order
        if (order.discountPercentage > 50) return false;
        return true;
    }
}

Both OrderPrinter and OrderValidator reach directly into Order's fields. They know that items are stored in a List, that discount is a percentage stored as a double, that customer is a direct reference. If Order changes its internal representation — say, wrapping items in an immutable collection, or changing discount from a percentage to a Discount object — both classes break.

What principle does it break?

Encapsulation. The entire point of encapsulation is that a class hides its internals behind a public interface. Inappropriate intimacy violates this by letting other classes depend on details they shouldn't see. The class boundary exists on paper but not in practice.

It also violates the Law of Demeter (talk to your friends, not to your friends' friends): order.customer.name chains through two objects, creating a dependency on both Order's internal structure and Customer's fields.

How to fix it

Hide Delegate — expose behavior, not internals

Make Order responsible for answering questions about itself. Other classes ask, they don't reach in.

public class Order {
    private final List<OrderItem> items = new ArrayList<>();
    private double discountPercentage;
    private Customer customer;

    public void addItem(OrderItem item) {
        items.add(item);
    }

    public List<OrderItem> getItems() {
        return Collections.unmodifiableList(items);
    }

    public String getCustomerName() {
        return customer.getName();
    }

    public double getDiscountedTotal() {
        double total = items.stream()
            .mapToDouble(OrderItem::getLineTotal)
            .sum();
        return total * (1 - discountPercentage / 100);
    }

    public double getDiscountedLineTotal(OrderItem item) {
        double lineTotal = item.getLineTotal();
        if (discountPercentage > 0) {
            lineTotal *= (1 - discountPercentage / 100);
        }
        return lineTotal;
    }

    public boolean hasItems() { return !items.isEmpty(); }
    public boolean hasCustomer() { return customer != null; }
    public boolean isDiscountReasonable() { return discountPercentage <= 50; }
}

Now OrderPrinter and OrderValidator use the public interface:

public class OrderPrinter {
    public String print(Order order) {
        StringBuilder sb = new StringBuilder();
        sb.append("Customer: ").append(order.getCustomerName()).append("\n");

        for (OrderItem item : order.getItems()) {
            double lineTotal = order.getDiscountedLineTotal(item);
            sb.append(item.getName()).append(": $")
              .append(String.format("%.2f", lineTotal)).append("\n");
        }

        sb.append("Total: $")
          .append(String.format("%.2f", order.getDiscountedTotal()));
        return sb.toString();
    }
}

public class OrderValidator {
    public boolean validate(Order order) {
        if (!order.hasItems()) return false;
        if (!order.hasCustomer()) return false;
        if (!order.isDiscountReasonable()) return false;

        for (OrderItem item : order.getItems()) {
            if (!item.isValid()) return false;
        }
        return true;
    }
}

Neither class knows how Order stores its items, how discounts are represented, or how the customer is linked. Order can change its internals freely.

Move Method — move behavior to the class that owns the data

When a method in one class does most of its work with another class's data, move it. The total calculation in OrderPrinter belongs on Order — and we've already moved it there as getDiscountedTotal().

Extract Class — when two classes are too intimate, merge the shared concern

If two classes are tightly intertwined because they share a concept, extract that concept into its own class that both depend on through a clean interface.
