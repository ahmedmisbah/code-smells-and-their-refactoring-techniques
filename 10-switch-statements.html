Switch Statements

Also known as: Type Code Switching

Description

A switch statement (or cascading if/else if) that dispatches behavior based on the type or category of an object. The problem isn't that switch exists — it's a fine construct for simple value mapping. The smell arises when the same switch on the same type code appears in multiple places throughout the codebase. Add a new type and you're hunting down every switch that inspects it, updating each one, and hoping you didn't miss any.

The deeper issue is that the code is doing manually what polymorphism does automatically. Instead of asking "what type are you?" and branching, you let each type carry its own behavior. The object knows how to do its thing — you just ask it.

Example

public class PayrollCalculator {

    public double calculatePay(Employee employee) {
        switch (employee.getType()) {
            case "full_time":
                return employee.getSalary();
            case "part_time":
                return employee.getHourlyRate() * employee.getHoursWorked();
            case "contractor":
                return employee.getDailyRate() * employee.getDaysWorked();
            case "intern":
                return employee.getStipend();
            default:
                throw new IllegalArgumentException("Unknown type: " + employee.getType());
        }
    }

    public double calculateBonus(Employee employee) {
        switch (employee.getType()) {
            case "full_time":
                return employee.getSalary() * 0.15;
            case "part_time":
                return employee.getHourlyRate() * employee.getHoursWorked() * 0.05;
            case "contractor":
                return 0;
            case "intern":
                return 500;
            default:
                throw new IllegalArgumentException("Unknown type: " + employee.getType());
        }
    }

    public int getVacationDays(Employee employee) {
        switch (employee.getType()) {
            case "full_time":  return 20;
            case "part_time":  return 10;
            case "contractor": return 0;
            case "intern":     return 5;
            default:
                throw new IllegalArgumentException("Unknown type: " + employee.getType());
        }
    }
}

Three methods, same four-branch switch on the same type string. If the company adds a "freelance" type, you update three methods (and every other switch on getType() scattered across the codebase). Miss one and you get a runtime exception.

What principle does it break?

Open/Closed Principle (OCP). Adding a new type should mean adding new code, not modifying existing code in multiple places. Every switch that dispatches on type is a point that must be edited when a type is added — the system is open for modification, closed for extension, exactly backwards.

It also violates Tell, Don't Ask: instead of telling the object what to do and letting it decide how, the code interrogates the object's type and makes the decision externally.

How to fix it

Replace Conditional with Polymorphism

The classic fix: turn each branch into a subclass that overrides a method.

public abstract class Employee {
    private String name;

    public abstract double calculatePay();
    public abstract double calculateBonus();
    public abstract int getVacationDays();
}

public class FullTimeEmployee extends Employee {
    private double salary;

    @Override
    public double calculatePay() { return salary; }

    @Override
    public double calculateBonus() { return salary * 0.15; }

    @Override
    public int getVacationDays() { return 20; }
}

public class PartTimeEmployee extends Employee {
    private double hourlyRate;
    private int hoursWorked;

    @Override
    public double calculatePay() { return hourlyRate * hoursWorked; }

    @Override
    public double calculateBonus() { return calculatePay() * 0.05; }

    @Override
    public int getVacationDays() { return 10; }
}

public class Contractor extends Employee {
    private double dailyRate;
    private int daysWorked;

    @Override
    public double calculatePay() { return dailyRate * daysWorked; }

    @Override
    public double calculateBonus() { return 0; }

    @Override
    public int getVacationDays() { return 0; }
}

public class Intern extends Employee {
    private double stipend;

    @Override
    public double calculatePay() { return stipend; }

    @Override
    public double calculateBonus() { return 500; }

    @Override
    public int getVacationDays() { return 5; }
}
public class PayrollCalculator {
    public double calculatePay(Employee employee) {
        return employee.calculatePay();
    }

    public double calculateBonus(Employee employee) {
        return employee.calculateBonus();
    }

    public int getVacationDays(Employee employee) {
        return employee.getVacationDays();
    }
}

Adding a Freelancer type means writing one new class. Zero existing code changes. Every switch disappears.

Replace Type Code with Strategy

When the type can change at runtime, or when subclassing the host isn't practical, extract the varying behavior into a strategy interface.

Before — shipping calculation switches on delivery type:

public class Order {
    private String deliveryType; // "standard", "express", "overnight"

    public double getShippingCost(double weight) {
        switch (deliveryType) {
            case "standard":  return weight * 0.50;
            case "express":   return weight * 1.50 + 5.00;
            case "overnight": return weight * 3.00 + 15.00;
            default: throw new IllegalArgumentException("Unknown: " + deliveryType);
        }
    }
}

After — a strategy per delivery type:

public interface ShippingStrategy {
    double calculate(double weight);
}

public class StandardShipping implements ShippingStrategy {
    public double calculate(double weight) { return weight * 0.50; }
}

public class ExpressShipping implements ShippingStrategy {
    public double calculate(double weight) { return weight * 1.50 + 5.00; }
}

public class OvernightShipping implements ShippingStrategy {
    public double calculate(double weight) { return weight * 3.00 + 15.00; }
}

public class Order {
    private ShippingStrategy shipping;

    public void setShipping(ShippingStrategy shipping) {
        this.shipping = shipping;
    }

    public double getShippingCost(double weight) {
        return shipping.calculate(weight);
    }
}

The Order can change its shipping strategy at runtime. The switch is gone. A new delivery option is one new class.

Replace Conditional with Map Lookup

When the switch is doing simple value mapping — input to output, or input to a short action — a Map does the same job with less syntax, no fall-through risk, and the ability to be built dynamically.

Before — a switch that maps states to tax rates:

public double getTaxRate(String state) {
    switch (state) {
        case "CA": return 0.0975;
        case "NY": return 0.08875;
        case "TX": return 0.0625;
        default:   return 0.07;
    }
}

After — a map lookup:

private static final Map<String, Double> TAX_RATES = Map.of(
    "CA", 0.0975,
    "NY", 0.08875,
    "TX", 0.0625
);

public double getTaxRate(String state) {
    return TAX_RATES.getOrDefault(state, 0.07);
}

It also works for mapping to behavior using Function:

private final Map<String, Function<Employee, Double>> bonusStrategies = Map.of(
    "full_time",  emp -> emp.getSalary() * 0.15,
    "part_time",  emp -> emp.getHourlyRate() * emp.getHoursWorked() * 0.05,
    "contractor", emp -> 0.0,
    "intern",     emp -> 500.0
);

public double calculateBonus(Employee emp) {
    return bonusStrategies
        .getOrDefault(emp.getType(), e -> 0.0)
        .apply(emp);
}

Where it shines: simple value-to-value or value-to-lambda mappings, especially when the entries might come from configuration or need to be modified at runtime. It's lighter than a full polymorphism hierarchy when the logic per branch is a one-liner.

Where it doesn't replace polymorphism: when each branch involves complex, multi-method behavior (pay calculation + bonus + vacation + benefits), cramming all of that into lambdas in a map gets ugly fast. That's where subclasses or strategy objects are the better tool.

When a switch is fine

Not every switch is a smell. A switch is perfectly appropriate when it occurs in exactly one place — typically a factory that maps a value to an object:

public class EmployeeFactory {
    public static Employee create(String type) {
        switch (type) {
            case "full_time":  return new FullTimeEmployee();
            case "part_time":  return new PartTimeEmployee();
            case "contractor": return new Contractor();
            case "intern":     return new Intern();
            default: throw new IllegalArgumentException("Unknown type: " + type);
        }
    }
}

This single switch is the only place in the system that knows about type strings. After this point, everything works through polymorphism. One switch in a factory is fine — it's the duplicated switches that are the smell.
