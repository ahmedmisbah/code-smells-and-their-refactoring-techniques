Speculative Generality

Description

Code that was written to handle future requirements that never materialized. Abstract classes with only one subclass. Interfaces implemented by a single class. Parameters that are never used in any real call. Hook methods that nobody overrides. Entire frameworks built to support flexibility that nobody asked for.

The developer thought "someday we might need to…" and built the infrastructure in advance. That someday never came, but the infrastructure remains — adding complexity, requiring maintenance, and confusing every new developer who wonders what it's for.

Example

public abstract class AbstractNotificationDispatcher<T extends Notification> {
    protected abstract NotificationChannel<T> getChannel();
    protected abstract NotificationFormatter<T> getFormatter();
    protected abstract NotificationValidator<T> getValidator();
    protected abstract RetryPolicy getRetryPolicy();

    public final void dispatch(T notification) {
        if (!getValidator().isValid(notification)) {
            throw new InvalidNotificationException(notification);
        }
        String formatted = getFormatter().format(notification);
        RetryPolicy policy = getRetryPolicy();
        int attempts = 0;
        while (attempts < policy.getMaxRetries()) {
            try {
                getChannel().send(formatted);
                return;
            } catch (ChannelException e) {
                attempts++;
                if (attempts >= policy.getMaxRetries()) throw e;
                policy.waitBeforeRetry(attempts);
            }
        }
    }
}

There's only one subclass:

public class EmailDispatcher extends AbstractNotificationDispatcher<EmailNotification> {
    protected NotificationChannel<EmailNotification> getChannel() {
        return new SmtpChannel();
    }
    protected NotificationFormatter<EmailNotification> getFormatter() {
        return new EmailFormatter();
    }
    protected NotificationValidator<EmailNotification> getValidator() {
        return new EmailValidator();
    }
    protected RetryPolicy getRetryPolicy() {
        return new SimpleRetryPolicy(3);
    }
}

An abstract class, four abstract methods, generics, a retry framework, custom exceptions — all to support one email sender. The "what if we add SMS, push, and Slack?" never happened. The entire abstract hierarchy is speculative.

What principle does it break?

YAGNI (You Aren't Gonna Need It). Build what you need now. Don't build infrastructure for requirements you don't have. Speculative code is dead weight with a maintenance cost — it must be read, understood, tested, and updated even though it serves no current purpose.

It also violates simplicity: the simplest solution that works is always preferable. An abstract framework for one implementation is never the simplest solution.

How to fix it

Collapse Hierarchy

When an abstract class has only one subclass, merge them.

After — the abstract class is gone:

public class EmailDispatcher {
    private final SmtpChannel channel = new SmtpChannel();
    private final EmailFormatter formatter = new EmailFormatter();
    private final EmailValidator validator = new EmailValidator();
    private static final int MAX_RETRIES = 3;

    public void dispatch(EmailNotification notification) {
        if (!validator.isValid(notification)) {
            throw new IllegalArgumentException("Invalid notification");
        }
        String formatted = formatter.format(notification);
        for (int attempt = 1; attempt <= MAX_RETRIES; attempt++) {
            try {
                channel.send(formatted);
                return;
            } catch (ChannelException e) {
                if (attempt == MAX_RETRIES) throw e;
            }
        }
    }
}

If SMS support is genuinely needed later, extract the interface at that point — driven by actual requirements, not speculation.

Remove unused parameters, methods, and interfaces

Unused parameter: remove it from the signature.
Abstract method nobody overrides differently: make it concrete.
Interface with one implementor: inline it (unless it's needed for testing or dependency injection).
Hook method nobody hooks into: remove it.
