Incomplete Library Class

Description

A third-party library class that almost does what you need — but is missing a method, has a rigid API, or behaves slightly wrong for your use case. You can't modify the library class (you don't own the source, or it would be overwritten on the next update), so you work around it. The workarounds accumulate: utility classes full of static methods that take the library class as their first parameter, subclasses that override one method, adapter wrappers that translate the API.

Every library is incomplete from someone's perspective. The smell isn't that the library is bad — it's that the workaround code is scattered, duplicated, and awkward. The library gap creates friction throughout the codebase.

Example

Suppose you use a DateRange class from a library, but it lacks an overlap check and a duration method that your code needs repeatedly:

// Library class — you can't modify this
public class DateRange {
    private final LocalDate start;
    private final LocalDate end;

    public DateRange(LocalDate start, LocalDate end) {
        this.start = start;
        this.end = end;
    }

    public LocalDate getStart() { return start; }
    public LocalDate getEnd() { return end; }
    public boolean contains(LocalDate date) {
        return !date.isBefore(start) && !date.isAfter(end);
    }
}

The workarounds start:

// Caller 1 — inline overlap check
public class BookingService {
    public boolean hasConflict(DateRange booking, DateRange requested) {
        return !booking.getEnd().isBefore(requested.getStart())
            && !booking.getStart().isAfter(requested.getEnd());
    }
}

// Caller 2 — same logic, different implementation
public class ScheduleValidator {
    public boolean overlaps(DateRange a, DateRange b) {
        return a.getStart().isBefore(b.getEnd())
            && b.getStart().isBefore(a.getEnd());
    }
}

// Caller 3 — needs duration, computes it inline
public class ReportService {
    public long getDays(DateRange range) {
        return ChronoUnit.DAYS.between(range.getStart(), range.getEnd());
    }
}

// Caller 4 — also needs duration, computes it differently
public class BillingService {
    public int calculateDays(DateRange period) {
        return (int) (period.getEnd().toEpochDay() - period.getStart().toEpochDay());
    }
}

Four classes, two implementations of overlap checking (with subtly different boundary behavior), two implementations of duration calculation. The missing library methods have caused duplication and inconsistency across the codebase.

What principle does it break?

DRY. The workarounds duplicate the same concept — overlap checking, duration calculation — because the library doesn't provide them. Without a centralized solution, each caller invents its own, leading to inconsistent behavior and scattered logic.

How to fix it

Introduce Foreign Method — add a utility method on your side

When you need just one or two methods, add them as static helpers in a utility class:

public class DateRangeUtils {

    public static boolean overlaps(DateRange a, DateRange b) {
        return !a.getEnd().isBefore(b.getStart())
            && !a.getStart().isAfter(b.getEnd());
    }

    public static long durationInDays(DateRange range) {
        return ChronoUnit.DAYS.between(range.getStart(), range.getEnd());
    }
}

All callers use the same utility:

public class BookingService {
    public boolean hasConflict(DateRange booking, DateRange requested) {
        return DateRangeUtils.overlaps(booking, requested);
    }
}

One implementation, one place to fix boundary behavior, one place to test.

Introduce Local Extension — wrap or extend the library class

When the gap is larger — many missing methods, or the library class needs adaptation — create a local extension that wraps it:

public class EnhancedDateRange {
    private final DateRange delegate;

    public EnhancedDateRange(LocalDate start, LocalDate end) {
        this.delegate = new DateRange(start, end);
    }

    public EnhancedDateRange(DateRange range) {
        this.delegate = range;
    }

    // delegate existing methods
    public LocalDate getStart() { return delegate.getStart(); }
    public LocalDate getEnd() { return delegate.getEnd(); }
    public boolean contains(LocalDate date) { return delegate.contains(date); }

    // add missing methods
    public boolean overlaps(EnhancedDateRange other) {
        return !this.getEnd().isBefore(other.getStart())
            && !this.getStart().isAfter(other.getEnd());
    }

    public long durationInDays() {
        return ChronoUnit.DAYS.between(getStart(), getEnd());
    }

    public EnhancedDateRange intersection(EnhancedDateRange other) {
        if (!this.overlaps(other)) return null;
        LocalDate start = this.getStart().isAfter(other.getStart())
            ? this.getStart() : other.getStart();
        LocalDate end = this.getEnd().isBefore(other.getEnd())
            ? this.getEnd() : other.getEnd();
        return new EnhancedDateRange(start, end);
    }

    // convert back to library type when needed
    public DateRange toDateRange() { return delegate; }
}

The wrapper adds the missing behavior while preserving compatibility with the library type. Callers use EnhancedDateRange throughout the codebase, and the library gap is filled in one place.

Choosing between Foreign Method and Local Extension

Foreign Method (utility class): best when you need 1–3 extra methods and the library class is used everywhere. Low ceremony, easy to add.
Local Extension (wrapper class): best when you need many extra methods, or when you want the enriched type to flow through your codebase as a first-class citizen. Higher upfront cost, but cleaner long-term.
