Oddball Solution

Also known as: Black Sheep

Description

A problem that is solved in two (or more) different ways within the same codebase. One module parses dates with SimpleDateFormat, another uses DateTimeFormatter, and a third uses a hand-rolled regex. One controller validates input with annotations, another uses an if/else chain, and a third calls a validation service. The solutions aren't wrong individually — they're inconsistent collectively.

The inconsistency confuses developers ("which approach should I use for my new code?"), makes patterns harder to find ("I'll search for all date parsing — oh wait, there are three different patterns"), and multiplies the maintenance burden (three approaches to update if the date format changes).

The "oddball" is the one that diverges from the established convention — the black sheep among white sheep. But sometimes there's no established convention at all, just multiple ad-hoc approaches.

Example

// Approach 1: parsing config values with a helper class
public class ConfigParser {
    public int getTimeout() {
        return Integer.parseInt(properties.getProperty("timeout", "5000"));
    }

    public boolean isDebugMode() {
        return Boolean.parseBoolean(properties.getProperty("debug", "false"));
    }
}

// Approach 2: same problem, different class, inline parsing
public class NotificationSettings {
    private final Properties props;

    public int getRetryLimit() {
        String val = props.getProperty("notification.retries");
        if (val == null || val.trim().isEmpty()) return 3;
        try {
            return Integer.parseInt(val.trim());
        } catch (NumberFormatException e) {
            return 3;
        }
    }
}

// Approach 3: same problem, yet another approach with Optional
public class CacheConfig {
    private final Properties props;

    public int getMaxEntries() {
        return Optional.ofNullable(props.getProperty("cache.max"))
            .map(String::trim)
            .filter(s -> !s.isEmpty())
            .map(Integer::parseInt)
            .orElse(1000);
    }
}

Three classes, three approaches to "read an integer from a properties file with a default." One uses getProperty with a default string. Another manually handles null, empty, and parse errors. A third uses Optional chaining. None is wrong, but the inconsistency means three patterns to understand, three patterns to maintain, and no clear precedent for the next developer.

What principle does it break?

Consistency / DRY. When the same problem is solved differently in different places, the codebase lacks a single canonical approach. Developers waste time choosing between approaches, and the codebase drifts further apart over time. It's a form of duplication — not of code, but of decisions.

How to fix it

Unify into a consistent approach

Pick the best solution and standardize on it. Extract it into a shared utility or pattern that all callers use.

public class ConfigUtils {
    private final Properties props;

    public ConfigUtils(Properties props) {
        this.props = props;
    }

    public int getInt(String key, int defaultValue) {
        String val = props.getProperty(key);
        if (val == null || val.trim().isEmpty()) return defaultValue;
        try {
            return Integer.parseInt(val.trim());
        } catch (NumberFormatException e) {
            return defaultValue;
        }
    }

    public boolean getBoolean(String key, boolean defaultValue) {
        String val = props.getProperty(key);
        if (val == null || val.trim().isEmpty()) return defaultValue;
        return Boolean.parseBoolean(val.trim());
    }

    public String getString(String key, String defaultValue) {
        String val = props.getProperty(key);
        if (val == null || val.trim().isEmpty()) return defaultValue;
        return val.trim();
    }
}

All three classes now use the same approach:

public class ConfigParser {
    private final ConfigUtils config;

    public int getTimeout() { return config.getInt("timeout", 5000); }
    public boolean isDebugMode() { return config.getBoolean("debug", false); }
}

public class NotificationSettings {
    private final ConfigUtils config;

    public int getRetryLimit() { return config.getInt("notification.retries", 3); }
}

public class CacheConfig {
    private final ConfigUtils config;

    public int getMaxEntries() { return config.getInt("cache.max", 1000); }
}

One pattern. One utility. Every config read looks the same, behaves the same, and handles edge cases the same way. The next developer has a clear precedent to follow.

Substitute Algorithm

When one solution is clearly better than the others (more robust, more readable, better error handling), replace the inferior ones with the superior one. Don't compromise — pick the best and make it the standard.
