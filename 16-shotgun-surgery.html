Shotgun Surgery

Description

One logical change requires small edits scattered across many different classes. You add a new field to the domain — say, a customer's middle name — and suddenly you're updating the entity class, the DTO, the mapper, the validator, the repository query, the API response, the report template, and the test fixtures. You make the same conceptual change in fifteen places, and if you miss one, things silently break.

Shotgun Surgery is the mirror image of Divergent Change. Divergent Change means one class changes for many reasons. Shotgun Surgery means one reason causes changes in many classes. They're the two sides of misplaced responsibility.

The smell is a sign that a single responsibility has been scattered — bits of it live in many places instead of being consolidated in one.

Example

Suppose you need to add support for a new currency. Here's what you have to touch:

// 1. The enum
public enum Currency {
    USD, EUR, GBP  // add JPY here
}

// 2. The formatter
public class CurrencyFormatter {
    public String format(double amount, Currency currency) {
        switch (currency) {
            case USD: return "$" + amount;
            case EUR: return "€" + amount;
            case GBP: return "£" + amount;
            // add JPY case here
            default: throw new IllegalArgumentException();
        }
    }
}

// 3. The exchange rate service
public class ExchangeRateService {
    public double getRate(Currency from, Currency to) {
        if (from == Currency.USD && to == Currency.EUR) return 0.93;
        if (from == Currency.USD && to == Currency.GBP) return 0.79;
        // add USD→JPY, EUR→JPY, GBP→JPY here
        return 1.0;
    }
}

// 4. The validator
public class PaymentValidator {
    private static final Set<Currency> SUPPORTED = Set.of(
        Currency.USD, Currency.EUR, Currency.GBP
        // add JPY here
    );

    public boolean isSupported(Currency currency) {
        return SUPPORTED.contains(currency);
    }
}

// 5. The report generator
public class FinancialReport {
    public String generate() {
        // hardcoded column for each currency
        return "USD | EUR | GBP";  // add | JPY here
    }
}

// 6. The test data
public class TestFixtures {
    public static Payment usdPayment() { return new Payment(100, Currency.USD); }
    public static Payment eurPayment() { return new Payment(100, Currency.EUR); }
    public static Payment gbpPayment() { return new Payment(100, Currency.GBP); }
    // add jpyPayment() here
}

Adding one currency touches six files. Each edit is small, but missing any one of them (especially the validator or the exchange rate service) means silent bugs. And six files is generous — a real system might require twenty or more edits.

What principle does it break?

Cohesion. Everything that defines "what it means to support a currency" is scattered. The symbol lives in the formatter, the exchange rates live in the rate service, the validation lives in the validator, and the test data lives in the fixtures. There's no single place that represents "a currency and everything about it."

This is also a violation of the DRY principle at a conceptual level: the knowledge of "which currencies exist and how they behave" is repeated in fragments across the codebase.

How to fix it

Move Method + Move Field — consolidate the scattered pieces

Pull currency-specific knowledge into the Currency type itself:

public enum Currency {
    USD("$", "US Dollar"),
    EUR("€", "Euro"),
    GBP("£", "British Pound"),
    JPY("¥", "Japanese Yen");   // add here — once

    private final String symbol;
    private final String displayName;

    Currency(String symbol, String displayName) {
        this.symbol = symbol;
        this.displayName = displayName;
    }

    public String format(double amount) {
        return symbol + String.format("%.2f", amount);
    }

    public String getSymbol() { return symbol; }
    public String getDisplayName() { return displayName; }
}

The formatter collapses:

public class CurrencyFormatter {
    public String format(double amount, Currency currency) {
        return currency.format(amount);
    }
}

The validator collapses — if it's in the enum, it's supported:

public class PaymentValidator {
    public boolean isSupported(Currency currency) {
        return currency != null; // if it parses, it's valid
    }
}

The exchange rate service can be driven by data rather than conditionals:

public class ExchangeRateService {
    private final Map<String, Double> rates; // loaded from config or database

    public ExchangeRateService(Map<String, Double> rates) {
        this.rates = rates;
    }

    public double getRate(Currency from, Currency to) {
        String key = from.name() + "_" + to.name();
        return rates.getOrDefault(key, 1.0);
    }
}

Now adding JPY means: add one enum constant, add exchange rate rows in the config data. One or two locations, not six.

Inline Class — when scatter is caused by over-decomposition

Sometimes Shotgun Surgery happens because a concept was split into too many tiny classes. If CurrencySymbol, CurrencyName, CurrencyValidator, and CurrencyFormatter are all separate classes with one method each, inlining them back into a single Currency class reduces the scatter.

The goal is a balance: each concept lives in one place (avoiding Shotgun Surgery) but each class has one responsibility (avoiding Divergent Change). When the pendulum swings too far toward decomposition, Inline Class pushes it back.
