Dead Code

Description

Code that is never executed — unreachable methods, unused variables, commented-out blocks, conditions that can never be true, parameters that are never passed, imports that are never used. Dead code is noise. It distracts readers, inflates the codebase, shows up in search results, and creates the nagging question: "is this used? Is it safe to delete? Was it left here for a reason?"

Dead code accumulates naturally. A method gets replaced but the old one isn't deleted. A feature flag is removed but the disabled path stays. A refactoring moves logic elsewhere but the original isn't cleaned up. Each piece is small, but together they make the codebase feel cluttered and untrustworthy.

Example

public class UserService {
    private static final int OLD_MAX_RETRIES = 5;  // unused constant

    public User findUser(String id) {
        return userRepository.findById(id);
    }

    // Old implementation — replaced by findUser above
    // public User getUser(String id) {
    //     User user = cache.get(id);
    //     if (user == null) {
    //         user = database.query("SELECT * FROM users WHERE id = ?", id);
    //         cache.put(id, user);
    //     }
    //     return user;
    // }

    public void deleteUser(String id) {
        User user = findUser(id);
        if (user == null) return;

        userRepository.delete(id);

        if (false) {  // dead condition — never executes
            auditLog.record("Deleted user: " + id);
        }
    }

    private void migrateFromLegacySystem() {
        // nobody calls this — leftover from 2019 migration
    }

    public void updateUser(String id, String name, String email,
                            String unused1, String unused2) {
        // unused1 and unused2 are never read
        userRepository.update(id, name, email);
    }
}

Unused constant, commented-out method, dead if (false) block, orphaned private method, unused parameters — five forms of dead code in one class.

What principle does it break?

Simplicity and signal-to-noise ratio. Every line of dead code is a line a reader must process and then dismiss. It clutters search results, complicates diffs, and makes the living code harder to find. It also creates false dependencies: tools that analyze code paths may include dead code in their reports, giving a misleading picture of the system's complexity.

How to fix it

Delete it. That's the entire fix. Dead code is in version control — if you ever need it back, git log has it. There's no reason to keep it in the living codebase.

Unused methods and fields: delete them. Your IDE or a static analysis tool can find them.
Commented-out code: delete it. If it's important, it's in version history.
Unreachable conditions (if (false)): delete the block.
Unused parameters: remove them from the signature and update callers.
Unused imports: delete them (most IDEs do this automatically).
public class UserService {

    public User findUser(String id) {
        return userRepository.findById(id);
    }

    public void deleteUser(String id) {
        User user = findUser(id);
        if (user == null) return;
        userRepository.delete(id);
    }

    public void updateUser(String id, String name, String email) {
        userRepository.update(id, name, email);
    }
}

Half the class is gone. What remains is what matters.
