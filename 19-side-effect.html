Side Effect

Description

A method that does something beyond its primary, advertised purpose — it performs hidden, secondary work that callers don't expect. The method's name says one thing, but its body does that thing plus something else that the caller can't see without reading the implementation. A query that also modifies state. A validation that also sets an error message. A check that also resizes a data structure.

The problem is surprise. When a caller invokes shouldGrow(), they expect a yes-or-no answer. If the method also expands the collection's capacity, the caller has no idea that an innocent request for information caused a state change. The method is lying by omission — its name is a partial truth.

Most side effects are harmful. Some are acceptable — an Iterator.next() that advances the cursor and returns the current element is a well-understood convention, and logging inside a method is generally tolerated. The smell applies when the secondary behavior is unrelated to the method's stated purpose and invisible to the caller.

Example

public class UserManager {
    private String errorMessage = "";
    private int failedAttempts = 0;

    public boolean accept(String username, String password) {
        String foundPassword = find(username);
        if (foundPassword == null || !foundPassword.equals(password)) {
            errorMessage = "Incorrect username or password. Please try again.";
            failedAttempts++;                  // side effect: tracks attempts
            if (failedAttempts >= 3) {
                lockAccount(username);         // side effect: locks account
            }
            return false;
        }
        errorMessage = "";
        failedAttempts = 0;
        updateLastLogin(username);             // side effect: updates login timestamp
        return true;
    }

    private String find(String username) { /* ... */ return null; }
    private void lockAccount(String username) { /* ... */ }
    private void updateLastLogin(String username) { /* ... */ }

    public String getErrorMessage() { return errorMessage; }
}

The method is called accept — its stated purpose is to determine whether a username and password are acceptable. But it also sets an error message field, increments a failed attempt counter, potentially locks the account, resets state on success, and updates the last login timestamp. A caller who just wants to check credentials gets five hidden behaviors for free.

This creates problems:

Testing is painful. You can't test credential validation without also triggering account locking, timestamp updates, and error message state.
Reuse is impossible. If another part of the system needs to validate credentials without locking accounts (say, a "verify current password before changing it" flow), it can't use accept without triggering the side effects.
Debugging is hard. When a user reports their account was locked, you trace it to accept, but the method name gives no hint that it locks anything.

Here's another example — a query that silently mutates:

public class ShoppingCart {
    private List<CartItem> items = new ArrayList<>();
    private boolean totaled = false;
    private double cachedTotal;

    public double getTotal() {
        items.removeIf(item -> item.getQuantity() <= 0);  // side effect: removes items
        cachedTotal = items.stream()
            .mapToDouble(i -> i.getPrice() * i.getQuantity())
            .sum();
        totaled = true;                                     // side effect: sets flag
        return cachedTotal;
    }
}

getTotal() looks like a pure query, but it silently removes invalid items and sets a flag. Calling it twice on the same cart can produce different results — items disappear between calls. A getter that modifies the object is a trap.

What principle does it break?

Command-Query Separation (CQS). A method should either be a command (changes state, returns nothing) or a query (returns a value, changes nothing). Side effects violate this by smuggling commands into queries. When getTotal() both returns a value and removes items, it's doing both — and the caller only expects one.

It also violates the Principle of Least Surprise: a method should do what its name says and nothing more. Hidden behavior undermines trust in the codebase — developers start reading every method's source before calling it, which defeats the purpose of abstraction.

Single Responsibility Principle (SRP) applies as well: the method is doing its stated job and unrelated jobs. Each side effect is a separate responsibility smuggled into a method that didn't ask for it.

How to fix it

Separate Commands from Queries

Split the method into pieces that each do one thing.

Before — accept validates and produces side effects:

public boolean accept(String username, String password) {
    String foundPassword = find(username);
    if (foundPassword == null || !foundPassword.equals(password)) {
        errorMessage = "Incorrect username or password. Please try again.";
        failedAttempts++;
        if (failedAttempts >= 3) {
            lockAccount(username);
        }
        return false;
    }
    errorMessage = "";
    failedAttempts = 0;
    updateLastLogin(username);
    return true;
}

After — validation is pure, side effects are explicit separate steps:

public boolean isValidCredentials(String username, String password) {
    String foundPassword = find(username);
    return foundPassword != null && foundPassword.equals(password);
}

public void recordFailedAttempt(String username) {
    failedAttempts++;
    if (failedAttempts >= 3) {
        lockAccount(username);
    }
}

public void recordSuccessfulLogin(String username) {
    failedAttempts = 0;
    updateLastLogin(username);
}

The caller now orchestrates explicitly:

if (userManager.isValidCredentials(username, password)) {
    userManager.recordSuccessfulLogin(username);
} else {
    userManager.recordFailedAttempt(username);
}

Every action is visible in the calling code. Nothing is hidden. isValidCredentials can be called from a "verify current password" flow without locking accounts. Each method is independently testable.

Make queries side-effect-free

For getters and queries, remove the mutation entirely.

Before — getTotal() silently removes items:

public double getTotal() {
    items.removeIf(item -> item.getQuantity() <= 0);
    cachedTotal = items.stream()
        .mapToDouble(i -> i.getPrice() * i.getQuantity())
        .sum();
    totaled = true;
    return cachedTotal;
}

After — the query computes without mutating; cleanup is a separate command:

public double getTotal() {
    return items.stream()
        .filter(item -> item.getQuantity() > 0)
        .mapToDouble(i -> i.getPrice() * i.getQuantity())
        .sum();
}

public void removeInvalidItems() {
    items.removeIf(item -> item.getQuantity() <= 0);
}

getTotal() is now pure — call it ten times and get the same result, with no state changes. The cleanup is a deliberate, visible command.

Use return values instead of mutating shared state

When a method needs to communicate a result and an error, return a result object instead of silently setting a field.

Before — accept communicates failure by setting errorMessage:

public boolean accept(String username, String password) {
    // ... sets this.errorMessage as a side effect
}

After — return a result that carries both the answer and the detail:

public class AuthResult {
    private final boolean success;
    private final String message;

    private AuthResult(boolean success, String message) {
        this.success = success;
        this.message = message;
    }

    public static AuthResult success() {
        return new AuthResult(true, "");
    }

    public static AuthResult failure(String reason) {
        return new AuthResult(false, reason);
    }

    public boolean isSuccess() { return success; }
    public String getMessage() { return message; }
}

public AuthResult authenticate(String username, String password) {
    String foundPassword = find(username);
    if (foundPassword == null || !foundPassword.equals(password)) {
        return AuthResult.failure("Incorrect username or password. Please try again.");
    }
    return AuthResult.success();
}

No shared mutable state. The error message travels with the result, not through a hidden field. The method is pure — no side effects at all.

When side effects are acceptable

Not every side effect is a smell. Some are well-established conventions:

Iterator.next() advances the cursor and returns the current element. The dual behavior is the entire point of the method, and it's universally understood.
Logging inside a method is a side effect (it writes to a file or console), but it's non-functional — it doesn't affect the method's return value or the caller's state. Most teams accept it.
Caching a computed result for performance (memoization) is a side effect that doesn't change observable behavior — the method returns the same answer, just faster the second time.

The key test: is the side effect surprising? If a reasonable developer reading only the method name and signature would not expect the secondary behavior, it's a smell. If the side effect is the method's known contract (like next()) or is non-functional (like logging), it's fine.

Innovations like Aspect-Oriented Programming (AOP) provide ways to separate cross-cutting concerns like logging, auditing, and security from business logic entirely — so even the "acceptable" side effects can be extracted when a cleaner separation is desired.
