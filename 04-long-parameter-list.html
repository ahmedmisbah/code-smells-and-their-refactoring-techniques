Long Parameter List

Description

A method that takes so many parameters that calling it becomes an exercise in reading documentation — or worse, guessing. Three parameters are comfortable; four is suspicious; five or more is almost always a sign that something has gone wrong. Long parameter lists make call sites unreadable, make it easy to swap arguments of the same type, and tightly couple the caller to the method's internal needs.

The smell often appears when a method asks for individual pieces of data that could be fetched from an object the caller already has, or when several parameters always appear together but haven't been recognized as a concept yet.

Example

public class ReportGenerator {

    public String generateReport(String title,
                                  String authorFirstName,
                                  String authorLastName,
                                  String authorEmail,
                                  String department,
                                  LocalDate startDate,
                                  LocalDate endDate,
                                  String timezone,
                                  boolean includeCharts,
                                  boolean includeSummary,
                                  String outputFormat,
                                  int maxPages) {

        String author = authorFirstName + " " + authorLastName;
        // ... hundreds of lines using all twelve parameters
        return "report";
    }
}

Every call site looks like this:

generator.generateReport(
    "Q3 Sales",
    "Ahmed", "Mahmoud", "ahmed@example.com",
    "Engineering",
    LocalDate.of(2025, 7, 1),
    LocalDate.of(2025, 9, 30),
    "UTC",
    true,
    false,
    "PDF",
    50
);

Twelve arguments, four of them String, two of them boolean. A reader at the call site has no idea what true, false means without jumping to the method signature. And if someone swaps "Ahmed" and "Mahmoud", the compiler won't say a word.

What principle does it break?

Encapsulation and abstraction. The method forces callers to know its internal data needs in granular detail. Instead of asking for a high-level concept ("who's the author?", "what period?"), it asks for raw pieces — first name, last name, email, start date, end date, timezone — demanding the caller assemble and pass them individually. This is a failure to abstract.

It also violates the Interface Segregation Principle in spirit: callers are forced to provide information they may not care about. A caller who just wants a simple report still has to decide about charts, summaries, formats, and page limits.

How to fix it

Introduce Parameter Object

When several parameters always travel together, they represent a concept that deserves a name. Bundle them into an object.

Before:

public String generateReport(String title,
                              String authorFirstName,
                              String authorLastName,
                              String authorEmail,
                              String department,
                              LocalDate startDate,
                              LocalDate endDate,
                              String timezone,
                              boolean includeCharts,
                              boolean includeSummary,
                              String outputFormat,
                              int maxPages) {
    // ...
}

After — three parameter objects replace twelve primitives:

public class Author {
    private final String firstName;
    private final String lastName;
    private final String email;
    private final String department;

    public String getFullName() {
        return firstName + " " + lastName;
    }

    // constructor, getters...
}

public class DateRange {
    private final LocalDate start;
    private final LocalDate end;
    private final String timezone;

    public boolean contains(LocalDate date) {
        return !date.isBefore(start) && !date.isAfter(end);
    }

    public long getDays() {
        return ChronoUnit.DAYS.between(start, end);
    }

    // constructor, getters...
}

public class ReportOptions {
    private final boolean includeCharts;
    private final boolean includeSummary;
    private final String outputFormat;
    private final int maxPages;

    // constructor, getters...
}
public String generateReport(String title, Author author,
                              DateRange period, ReportOptions options) {
    // ...
}

The call site becomes readable:

generator.generateReport(
    "Q3 Sales",
    new Author("Ahmed", "Mahmoud", "ahmed@example.com", "Engineering"),
    new DateRange(LocalDate.of(2025, 7, 1), LocalDate.of(2025, 9, 30), "UTC"),
    new ReportOptions(true, false, "PDF", 50)
);

Four parameters, each named and typed. No ambiguity about what true and false mean.

Preserve Whole Object

When a method pulls several values out of an object just to pass them individually, pass the object itself instead.

Before — extracting fields only to pass them along:

public void sendNotification(User user) {
    String name = user.getName();
    String email = user.getEmail();
    String language = user.getPreferredLanguage();
    String timezone = user.getTimezone();
    emailService.send(name, email, language, timezone, "Welcome!");
}

After — pass the whole object:

public void sendNotification(User user) {
    emailService.send(user, "Welcome!");
}

// inside EmailService:
public void send(User recipient, String message) {
    String localized = localize(message, recipient.getPreferredLanguage());
    String scheduled = adjustForTimezone(recipient.getTimezone());
    // ... uses recipient.getName(), recipient.getEmail() directly
}

The parameter list shrinks from five to two, and if EmailService later needs the user's phone number for SMS fallback, no signatures need to change.

Replace Parameter with Method Call

When a parameter can be computed from another parameter already being passed, stop passing it — let the method compute it.

Before — caller computes a value the method could derive itself:

double basePrice = order.getBasePrice();
double discount = order.getDiscount();
double tax = taxService.calculate(basePrice);
invoice.generate(order, basePrice, discount, tax);

After — the method fetches what it needs:

invoice.generate(order, taxService);

// inside Invoice:
public void generate(Order order, TaxService taxService) {
    double basePrice = order.getBasePrice();
    double discount = order.getDiscount();
    double tax = taxService.calculate(basePrice);
    // ...
}

Three parameters disappear because the method can reach them through the objects it already has.

Builder Pattern (for optional/configuration-heavy parameters)

When a method has many optional flags and settings — especially booleans — a Builder gives each parameter a name at the call site without inflating the constructor.

Before — boolean flags with no readable meaning:

Report report = new Report("Q3 Sales", author, period,
                            true, false, true, "PDF", 50, false);

After — a Builder makes every choice explicit:

Report report = new Report.Builder("Q3 Sales", author, period)
    .includeCharts(true)
    .includeSummary(false)
    .outputFormat("PDF")
    .maxPages(50)
    .build();
public class Report {
    private final String title;
    private final Author author;
    private final DateRange period;
    private final boolean includeCharts;
    private final boolean includeSummary;
    private final String outputFormat;
    private final int maxPages;

    private Report(Builder builder) {
        this.title = builder.title;
        this.author = builder.author;
        this.period = builder.period;
        this.includeCharts = builder.includeCharts;
        this.includeSummary = builder.includeSummary;
        this.outputFormat = builder.outputFormat;
        this.maxPages = builder.maxPages;
    }

    public static class Builder {
        private final String title;
        private final Author author;
        private final DateRange period;
        private boolean includeCharts = false;
        private boolean includeSummary = true;
        private String outputFormat = "PDF";
        private int maxPages = 100;

        public Builder(String title, Author author, DateRange period) {
            this.title = title;
            this.author = author;
            this.period = period;
        }

        public Builder includeCharts(boolean val) {
            this.includeCharts = val; return this;
        }

        public Builder includeSummary(boolean val) {
            this.includeSummary = val; return this;
        }

        public Builder outputFormat(String val) {
            this.outputFormat = val; return this;
        }

        public Builder maxPages(int val) {
            this.maxPages = val; return this;
        }

        public Report build() {
            return new Report(this);
        }
    }
}

The Builder separates required parameters (in the constructor) from optional ones (fluent setters with sensible defaults). Every call site is self-documenting.
