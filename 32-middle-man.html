Middle Man

Description

A class that exists only to delegate to another class — every method just forwards the call without adding any logic, transformation, or coordination. The class is an empty middleman, adding a layer of indirection with no value.

Middle Man is the opposite extreme of Message Chains. Message Chains are too many intermediary objects to navigate through. The typical fix — Hide Delegate — adds delegating methods. But if you overdo it, every method on the class just calls through to someone else, and the class becomes a useless middle man. The cure for Message Chains, taken too far, becomes Middle Man.

Example

public class Customer {
    private final Address address;
    private final ContactInfo contact;
    private final Account account;

    public Customer(Address address, ContactInfo contact, Account account) {
        this.address = address;
        this.contact = contact;
        this.account = account;
    }

    // every method is pure delegation — Customer adds nothing

    public String getStreet() { return address.getStreet(); }
    public String getCity() { return address.getCity(); }
    public String getState() { return address.getState(); }
    public String getZipCode() { return address.getZipCode(); }
    public String getCountry() { return address.getCountry(); }

    public String getEmail() { return contact.getEmail(); }
    public String getPhone() { return contact.getPhone(); }
    public String getFax() { return contact.getFax(); }

    public double getBalance() { return account.getBalance(); }
    public void deposit(double amount) { account.deposit(amount); }
    public void withdraw(double amount) { account.withdraw(amount); }
    public List<Transaction> getTransactions() { return account.getTransactions(); }
}

Twelve methods, every one a single-line forwarder. Customer doesn't decide anything, transform anything, or coordinate anything. It's a pass-through shell. Callers might as well talk to Address, ContactInfo, and Account directly.

What principle does it break?

Simplicity and meaningful abstraction. A class should justify its existence by providing behavior, enforcing invariants, or offering a genuinely useful abstraction. A pure delegator provides none of these — it's a layer of indirection that adds cognitive overhead (more classes to understand) without adding value.

How to fix it

Remove Middle Man — let callers talk directly to the delegate

Expose the delegate and let callers access it.

public class Customer {
    private final String name;
    private final Address address;
    private final ContactInfo contact;
    private final Account account;

    public Customer(String name, Address address,
                     ContactInfo contact, Account account) {
        this.name = name;
        this.address = address;
        this.contact = contact;
        this.account = account;
    }

    public String getName() { return name; }
    public Address getAddress() { return address; }
    public ContactInfo getContact() { return contact; }
    public Account getAccount() { return account; }
}

Callers use customer.getAddress().getCity() instead of customer.getCity(). Yes, that's a message chain — but it's a short one (two levels), and it correctly reflects the domain structure: a customer has an address.

The balance between Message Chains and Middle Man

These two smells pull in opposite directions:

Message Chains say "don't reach through multiple objects — add delegation."
Middle Man says "don't delegate everything — let callers access the object."

The balance is: delegate when the class adds value (coordination, transformation, invariant enforcement, or hiding an unstable internal structure). Expose the delegate when the class adds nothing and the internal structure is stable and well-known.

A good heuristic: if more than half the methods in a class are single-line delegations, you probably have a middle man. If the delegation methods add validation, logging, access control, or combine results from multiple delegates, the class is earning its keep.
