Alternative Classes with Different Interfaces

Description

Two or more classes that do essentially the same thing but expose different method names, parameter orders, or return types. They're interchangeable in purpose but not in practice — you can't swap one for the other without rewriting the calling code. The duplication isn't in the implementation (they might be completely different internally); it's in the concept. They serve the same role but refuse to agree on how to be called.

This smell often appears when two developers solve the same problem independently, or when a class is written to replace an old one but nobody aligns the interfaces. It also appears across libraries: you use one PDF generator with render(doc) and another with generate(document, options) — same intent, different shape.

Example

public class EmailNotifier {
    public void sendAlert(String recipientEmail, String subject, String body) {
        // sends an email
    }
}

public class SmsNotifier {
    public void notify(String phoneNumber, String message) {
        // sends an SMS
    }
}

public class PushNotifier {
    public boolean pushMessage(String deviceToken, String title, String payload,
                                boolean highPriority) {
        // sends a push notification
        return true;
    }
}

All three classes serve the same purpose — deliver a notification to a user — but they have different method names (sendAlert, notify, pushMessage), different parameter shapes, and different return types. The calling code can't treat them uniformly:

public class AlertService {
    public void alertUser(User user, String message) {
        if (user.prefersEmail()) {
            emailNotifier.sendAlert(user.getEmail(), "Alert", message);
        } else if (user.prefersSms()) {
            smsNotifier.notify(user.getPhone(), message);
        } else {
            pushNotifier.pushMessage(user.getDeviceToken(), "Alert", message, true);
        }
    }
}

Every new channel requires a new branch. The code can't iterate over notifiers or swap one for another.

What principle does it break?

Liskov Substitution Principle (LSP) and polymorphism in general. Classes that serve the same role should be substitutable for each other. When they have different interfaces, substitution is impossible — the caller must know the concrete type and call the right method with the right arguments. This defeats the purpose of abstraction.

It also violates DRY at the design level: the concept of "send a notification" is repeated three times with no unifying abstraction.

How to fix it

Rename Methods + Extract Interface

Align the interfaces, then extract a common type.

public interface Notifier {
    void send(String recipient, String subject, String body);
}

public class EmailNotifier implements Notifier {
    @Override
    public void send(String recipient, String subject, String body) {
        // recipient is an email address
        // sends an email with subject and body
    }
}

public class SmsNotifier implements Notifier {
    @Override
    public void send(String recipient, String subject, String body) {
        // recipient is a phone number
        // concatenates subject and body into SMS (subject may be ignored)
    }
}

public class PushNotifier implements Notifier {
    @Override
    public void send(String recipient, String subject, String body) {
        // recipient is a device token
        // sends push with subject as title and body as payload
    }
}

Now the calling code is uniform:

public class AlertService {
    private final List<Notifier> notifiers;

    public AlertService(List<Notifier> notifiers) {
        this.notifiers = notifiers;
    }

    public void alertUser(User user, String message) {
        for (Notifier notifier : notifiers) {
            notifier.send(user.getContactFor(notifier), "Alert", message);
        }
    }
}

Adding a new channel means writing one class that implements Notifier. Zero changes to AlertService.

Move Methods to align behavior

Sometimes the classes don't just have different names — they have genuinely different capabilities scattered across them. Use Move Method to consolidate behavior so each class covers the same ground, then extract the interface.

Before — one class formats, the other doesn't:

public class PdfExporter {
    public byte[] export(Report report) { /* ... */ }
    public String getFormattedTitle(Report report) { /* ... */ }
}

public class CsvExporter {
    public byte[] generate(Report report) { /* ... */ }
    // no title formatting — caller has to do it manually
}

After — move getFormattedTitle logic into both, align names, extract interface:

public interface ReportExporter {
    byte[] export(Report report);
    String getFormattedTitle(Report report);
}

public class PdfExporter implements ReportExporter {
    @Override
    public byte[] export(Report report) { /* ... */ }

    @Override
    public String getFormattedTitle(Report report) { /* ... */ }
}

public class CsvExporter implements ReportExporter {
    @Override
    public byte[] export(Report report) { /* ... */ }

    @Override
    public String getFormattedTitle(Report report) {
        // now CsvExporter also provides formatted titles
        return report.getTitle().toUpperCase();
    }
}

Both classes now have the same shape. They're interchangeable through the ReportExporter interface.
