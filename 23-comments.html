Comments (Deodorant Comment)

Description

A comment that exists because the code it describes is unclear, complex, or poorly named — the comment is deodorant sprayed over a bad smell rather than a fix for the smell itself. The code should be refactored so the comment becomes unnecessary.

This doesn't mean all comments are bad. Comments that explain why (business context, regulatory reasons, non-obvious trade-offs) are valuable. The smell is comments that explain what or how — things the code should express on its own through naming, structure, and simplicity. If you need a comment to explain what a block of code does, the real fix is to extract that block into a well-named method.

Example

public class OrderProcessor {

    public double process(Order o) {
        // Check if the order is valid
        if (o == null) throw new IllegalArgumentException();
        if (o.getItems().isEmpty()) throw new IllegalArgumentException();
        for (OrderItem i : o.getItems()) {
            if (i.getQty() <= 0) throw new IllegalArgumentException();
        }

        // Calculate the subtotal by summing line prices
        double st = 0;
        for (OrderItem i : o.getItems()) {
            // Apply category discount
            double lp = i.getP() * i.getQty();
            if (i.getCat().equals("e")) {
                lp *= 0.95; // 5% off electronics
            } else if (i.getCat().equals("b")) {
                lp *= 0.90; // 10% off books
            }
            st += lp;
        }

        // Apply VIP discount
        if (o.getCust().isVip()) {
            st *= 0.90; // 10% VIP discount
        }

        // Calculate tax
        double t = st * 0.08; // 8% tax rate

        return st + t;
    }
}

Every comment explains what the next lines do — because the variable names (o, st, lp, i, t) and method names (getP, getCat, getQty) are cryptic. The comments are compensating for bad naming. If the code were clear, the comments would be redundant.

What principle does it break?

Self-documenting code / readability. Code should express its intent through its structure and naming. When a comment restates what the code does, it's a sign that the code isn't speaking for itself. Worse, comments rot: the code changes but the comment often doesn't, creating misleading documentation that's worse than no documentation.

How to fix it

Rename variables and methods, then delete the comments

public class OrderProcessor {

    public double process(Order order) {
        validateOrder(order);
        double subtotal = calculateSubtotal(order);
        subtotal = applyVipDiscount(order, subtotal);
        double tax = calculateTax(subtotal);
        return subtotal + tax;
    }

    private void validateOrder(Order order) {
        if (order == null) throw new IllegalArgumentException("Order is null");
        if (order.getItems().isEmpty()) throw new IllegalArgumentException("No items");
        for (OrderItem item : order.getItems()) {
            if (item.getQuantity() <= 0) {
                throw new IllegalArgumentException("Invalid quantity");
            }
        }
    }

    private double calculateSubtotal(Order order) {
        return order.getItems().stream()
            .mapToDouble(this::getDiscountedPrice)
            .sum();
    }

    private double getDiscountedPrice(OrderItem item) {
        double linePrice = item.getPrice() * item.getQuantity();
        if (item.getCategory().equals("electronics")) return linePrice * 0.95;
        if (item.getCategory().equals("books")) return linePrice * 0.90;
        return linePrice;
    }

    private double applyVipDiscount(Order order, double subtotal) {
        if (order.getCustomer().isVip()) return subtotal * 0.90;
        return subtotal;
    }

    private double calculateTax(double subtotal) {
        return subtotal * 0.08;
    }
}

Zero comments. The method names are the comments: validateOrder, calculateSubtotal, getDiscountedPrice, applyVipDiscount, calculateTax. Each method reads as a sentence.

When comments are valuable

Why, not what: // Using insertion sort here because the list is nearly sorted (benchmarked at 3x faster than quicksort for our data)
Regulatory/business context: // GDPR Article 17: must delete user data within 30 days of request
Warnings: // WARNING: this method is not thread-safe — see JIRA-1234
TODOs with context: // TODO: replace with batch API when vendor supports it (ETA Q3)
Public API documentation: Javadoc/JSDoc explaining contracts for library consumers.
