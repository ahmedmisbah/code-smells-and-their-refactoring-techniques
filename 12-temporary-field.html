Temporary Field

Description

A field in a class that is only set and used under certain conditions — it's populated in some code paths and left null (or zero, or empty) in others. The field exists on every instance, but it only means something some of the time. Readers encounter the field and have to wonder: is this always valid? When is it set? What happens if I read it when it hasn't been set?

Temporary fields usually appear when a complex algorithm needs to share data between its internal methods, and instead of passing that data as parameters or creating a helper object, the developer stashes it in instance fields. The fields become a hidden communication channel between methods — a channel that's invisible to callers and meaningless outside the algorithm's execution.

Example

public class DiscountCalculator {
    private List<OrderItem> items;
    private Customer customer;
    private double subtotal;
    private double loyaltyDiscount;
    private double volumeDiscount;
    private double seasonalDiscount;
    private boolean isHolidaySeason;

    public double calculateDiscount(List<OrderItem> items, Customer customer) {
        this.items = items;            // stash in field
        this.customer = customer;      // stash in field
        this.subtotal = 0;
        this.loyaltyDiscount = 0;
        this.volumeDiscount = 0;
        this.seasonalDiscount = 0;
        this.isHolidaySeason = checkHolidaySeason();

        computeSubtotal();
        computeLoyaltyDiscount();
        computeVolumeDiscount();
        computeSeasonalDiscount();

        return loyaltyDiscount + volumeDiscount + seasonalDiscount;
    }

    private void computeSubtotal() {
        for (OrderItem item : items) {          // reads from field
            subtotal += item.getPrice() * item.getQuantity();  // writes to field
        }
    }

    private void computeLoyaltyDiscount() {
        if (customer.getLoyaltyYears() > 5) {   // reads from field
            loyaltyDiscount = subtotal * 0.10;  // writes to field
        }
    }

    private void computeVolumeDiscount() {
        if (items.size() > 20) {                // reads from field
            volumeDiscount = subtotal * 0.05;   // writes to field
        }
    }

    private void computeSeasonalDiscount() {
        if (isHolidaySeason) {                  // reads from field
            seasonalDiscount = subtotal * 0.15; // writes to field
        }
    }

    private boolean checkHolidaySeason() { /* ... */ return false; }
}

Seven fields exist solely to pass data between calculateDiscount and its helper methods. Between calls to calculateDiscount, every one of those fields is meaningless. If someone calls computeLoyaltyDiscount without first calling computeSubtotal, the result is silently wrong. The fields create invisible temporal coupling between methods — they must be called in the right order, but nothing enforces that.

What principle does it break?

Encapsulation and class cohesion. Fields should represent the enduring state of an object — things that are meaningful for the object's entire lifetime. Temporary fields violate this by using the object as a scratch pad. The class appears to have state it doesn't really have, misleading anyone reading the class definition.

It also violates least surprise: a caller looking at the class sees seven fields and reasonably assumes they're all part of the object's meaningful state. They're not — they're implementation debris from one algorithm.

How to fix it

Replace Method with Method Object (Extract Class)

Move the algorithm and its temporary state into a dedicated class where the fields are the meaningful state — for the duration of the computation.

public class DiscountCalculator {
    public double calculateDiscount(List<OrderItem> items, Customer customer) {
        return new DiscountComputation(items, customer).compute();
    }
}

public class DiscountComputation {
    private final List<OrderItem> items;
    private final Customer customer;
    private double subtotal;
    private double loyaltyDiscount;
    private double volumeDiscount;
    private double seasonalDiscount;
    private final boolean isHolidaySeason;

    public DiscountComputation(List<OrderItem> items, Customer customer) {
        this.items = items;
        this.customer = customer;
        this.isHolidaySeason = checkHolidaySeason();
    }

    public double compute() {
        computeSubtotal();
        computeLoyaltyDiscount();
        computeVolumeDiscount();
        computeSeasonalDiscount();
        return loyaltyDiscount + volumeDiscount + seasonalDiscount;
    }

    private void computeSubtotal() {
        for (OrderItem item : items) {
            subtotal += item.getPrice() * item.getQuantity();
        }
    }

    private void computeLoyaltyDiscount() {
        if (customer.getLoyaltyYears() > 5) {
            loyaltyDiscount = subtotal * 0.10;
        }
    }

    private void computeVolumeDiscount() {
        if (items.size() > 20) {
            volumeDiscount = subtotal * 0.05;
        }
    }

    private void computeSeasonalDiscount() {
        if (isHolidaySeason) {
            seasonalDiscount = subtotal * 0.15;
        }
    }

    private boolean checkHolidaySeason() { /* ... */ return false; }
}

DiscountComputation is a short-lived object — created, used, discarded. Its fields are meaningful for its entire (brief) lifetime. The original DiscountCalculator is clean, with no leftover fields.

Introduce Null Object for conditionally present data

When a temporary field represents an optional collaborator (set in some cases, null in others), replace the null with a Null Object that does nothing.

Before — a logger field that's only sometimes set:

public class DataProcessor {
    private Logger logger; // null if logging isn't needed

    public void process(Data data) {
        if (logger != null) {
            logger.log("Processing: " + data.getId());
        }
        // ... process ...
        if (logger != null) {
            logger.log("Done: " + data.getId());
        }
    }
}

After — a Null Object eliminates the conditional:

public interface Logger {
    void log(String message);
}

public class ConsoleLogger implements Logger {
    public void log(String message) {
        System.out.println(message);
    }
}

public class NullLogger implements Logger {
    public void log(String message) {
        // do nothing
    }
}

public class DataProcessor {
    private final Logger logger;

    public DataProcessor(Logger logger) {
        this.logger = logger;
    }

    public DataProcessor() {
        this.logger = new NullLogger(); // never null
    }

    public void process(Data data) {
        logger.log("Processing: " + data.getId());
        // ... process ...
        logger.log("Done: " + data.getId());
    }
}

The logger field is always valid. No null checks, no temporary state, no conditional confusion.
