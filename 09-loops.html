Loops

Description

Using imperative loops (for, while) to process collections when a pipeline of higher-level operations (map, filter, reduce) would express the intent more clearly. The smell isn't that loops exist — they're fundamental — but that they're used as a default even when the code is really doing a well-known operation: filtering elements, transforming each element, accumulating a result, or finding a match.

An imperative loop interleaves the mechanism of iteration (initialize counter, check condition, increment) with the purpose (what you're actually doing with each element). A pipeline separates these: the operations state the purpose, and the framework handles the mechanism.

Fowler introduced this smell in his second edition, acknowledging that language support for pipelines (Java Streams, C# LINQ, JavaScript array methods, Python comprehensions) has matured enough that the pipeline is often the clearer choice.

This is the mildest smell in the Bloaters category. Not every loop is a smell — complex iteration logic, loops with multiple exit conditions, or performance-critical inner loops may be clearer or faster as explicit loops. The smell applies when a loop is doing something a single pipeline operation could express in one line.

Example

public class EmployeeReport {
    private List<Employee> employees;

    public List<String> getHighEarnerNames() {
        List<String> result = new ArrayList<>();
        for (Employee emp : employees) {
            if (emp.getSalary() > 100_000) {
                result.add(emp.getName());
            }
        }
        return result;
    }

    public double getTotalPayroll() {
        double total = 0;
        for (Employee emp : employees) {
            total += emp.getSalary();
        }
        return total;
    }

    public Employee findById(String id) {
        for (Employee emp : employees) {
            if (emp.getId().equals(id)) {
                return emp;
            }
        }
        return null;
    }

    public Map<String, List<Employee>> groupByDepartment() {
        Map<String, List<Employee>> groups = new HashMap<>();
        for (Employee emp : employees) {
            String dept = emp.getDepartment();
            if (!groups.containsKey(dept)) {
                groups.put(dept, new ArrayList<>());
            }
            groups.get(dept).add(emp);
        }
        return groups;
    }

    public boolean hasAnyIntern() {
        for (Employee emp : employees) {
            if (emp.getRole().equals("intern")) {
                return true;
            }
        }
        return false;
    }
}

Each method is a recognizable pattern — filter-and-collect, sum, find-first, group-by, any-match — buried under loop boilerplate. The reader has to mentally simulate the loop to discover the intent.

What principle does it break?

Declarative intent / readability. Imperative loops describe how to iterate; pipelines describe what you want. When the "what" is a standard operation (filter, map, reduce, find, group), expressing it imperatively adds noise and obscures meaning. The reader has to reverse-engineer the loop to understand its purpose, when a pipeline would state it directly.

It also mildly violates separation of concerns: the loop body mixes iteration control with business logic. Pipelines separate them cleanly.

How to fix it

The primary refactoring is Replace Loop with Pipeline: rewrite the loop as a chain of stream operations.

public class EmployeeReport {
    private List<Employee> employees;

    public List<String> getHighEarnerNames() {
        return employees.stream()
            .filter(emp -> emp.getSalary() > 100_000)
            .map(Employee::getName)
            .collect(Collectors.toList());
    }

    public double getTotalPayroll() {
        return employees.stream()
            .mapToDouble(Employee::getSalary)
            .sum();
    }

    public Optional<Employee> findById(String id) {
        return employees.stream()
            .filter(emp -> emp.getId().equals(id))
            .findFirst();
    }

    public Map<String, List<Employee>> groupByDepartment() {
        return employees.stream()
            .collect(Collectors.groupingBy(Employee::getDepartment));
    }

    public boolean hasAnyIntern() {
        return employees.stream()
            .anyMatch(emp -> emp.getRole().equals("intern"));
    }
}

Each method now reads as a single statement that declares its intent. filter + map + collect is immediately recognizable. groupingBy replaces seven lines of manual map management. anyMatch replaces the early-return loop. findFirst returns an Optional instead of a null — a bonus improvement.

When to keep the loop

Pipelines aren't always better. Keep an explicit loop when:

Complex control flow — multiple break/continue conditions, nested loops with interleaved state, or early exits based on accumulated results.
Performance is critical — and the loop avoids object allocation overhead that streams introduce. In hot paths processing millions of elements, a plain for loop can meaningfully outperform a stream pipeline.
Readability is genuinely better as a loop — some multi-step transformations with side effects read more naturally as sequential statements than as a chain.

The goal is clarity. If a pipeline reads like English and the loop reads like assembly, use the pipeline. If the pipeline becomes a tangled chain of flatMap-peek-reduce that needs a paragraph of comments, keep the loop.
