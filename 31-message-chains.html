Message Chains

Description

A series of calls chained together where each call returns an object that the next call is made on: order.getCustomer().getAddress().getCity().toUpperCase(). The caller depends on the entire chain of intermediate objects — if any link changes, the caller breaks.

The problem isn't the dot syntax — fluent APIs and stream pipelines chain deliberately and are fine. The problem is when the chain navigates a structure of domain objects, reaching deep into a graph to extract something buried several levels down. The caller is coupled to the exact shape of the object graph, not just to the object it directly holds.

Example

public class InvoiceGenerator {

    public String generateHeader(Order order) {
        // four levels deep
        String companyName = order.getCustomer()
                                   .getCompany()
                                   .getLegalEntity()
                                   .getRegisteredName();

        String billingCity = order.getCustomer()
                                  .getBillingAddress()
                                  .getCity();

        String taxId = order.getCustomer()
                            .getCompany()
                            .getLegalEntity()
                            .getTaxId();

        String currency = order.getPaymentTerms()
                               .getPreferredCurrency()
                               .getCode();

        return String.format("Invoice for %s (%s)\n%s\nTax ID: %s\nCurrency: %s",
            companyName, billingCity, billingCity, taxId, currency);
    }
}

InvoiceGenerator knows that an order has a customer, who has a company, which has a legal entity, which has a registered name and tax ID. It also knows the customer has a billing address with a city, and the order has payment terms with a preferred currency with a code. That's knowledge of five classes deep. If Company moves taxId to a different sub-object, InvoiceGenerator breaks — even though it has nothing to do with company structure.

What principle does it break?

Law of Demeter (Principle of Least Knowledge). A method should only talk to its immediate friends — the object itself, its fields, parameters passed to it, and objects it creates. Reaching through a chain of getX().getY().getZ() talks to strangers — objects the caller was never meant to know about.

It also creates structural coupling: the caller is coupled not to one class but to an entire graph of classes and their relationships.

How to fix it

Hide Delegate — introduce intermediary methods

Let each object answer questions about itself rather than exposing its internal structure.

public class Order {
    private Customer customer;
    private PaymentTerms paymentTerms;

    public String getCustomerCompanyName() {
        return customer.getCompanyName();
    }

    public String getBillingCity() {
        return customer.getBillingCity();
    }

    public String getCustomerTaxId() {
        return customer.getTaxId();
    }

    public String getCurrencyCode() {
        return paymentTerms.getCurrencyCode();
    }
}

public class Customer {
    private Company company;
    private Address billingAddress;

    public String getCompanyName() {
        return company.getRegisteredName();
    }

    public String getBillingCity() {
        return billingAddress.getCity();
    }

    public String getTaxId() {
        return company.getTaxId();
    }
}

public class Company {
    private LegalEntity legalEntity;

    public String getRegisteredName() {
        return legalEntity.getRegisteredName();
    }

    public String getTaxId() {
        return legalEntity.getTaxId();
    }
}

public class PaymentTerms {
    private Currency preferredCurrency;

    public String getCurrencyCode() {
        return preferredCurrency.getCode();
    }
}

Now InvoiceGenerator talks only to Order:

public class InvoiceGenerator {
    public String generateHeader(Order order) {
        return String.format("Invoice for %s (%s)\n%s\nTax ID: %s\nCurrency: %s",
            order.getCustomerCompanyName(),
            order.getBillingCity(),
            order.getBillingCity(),
            order.getCustomerTaxId(),
            order.getCurrencyCode());
    }
}

One level deep. If the internal structure changes, only the delegation methods change — not the callers.

When chains are acceptable

Not all chains are message chains in the smell sense:

Fluent APIs — builder.setName("x").setAge(30).build() returns this at each step. The caller is coupled to one class, not a graph.
Stream pipelines — .stream().filter(...).map(...).collect(...) is a pipeline over one collection, not a navigation through a domain structure.
Configuration chains — config.database().connection().timeout() may be fine if the config hierarchy is stable and the chain represents a deliberate navigation API.

The smell applies specifically when the chain navigates through domain objects with different responsibilities.
