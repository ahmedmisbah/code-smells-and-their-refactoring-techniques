Large Class

Also known as: Blob

Description

A class that has grown to take on too many responsibilities — too many fields, too many methods, too many lines of code. Like long methods, large classes grow incrementally: it's always easier to add one more field or method to an existing class than to figure out where a new class should go. Over time, the class becomes a sprawling catch-all that knows too much and does too much.

Example

public class Employee {
    private String name;
    private String email;
    private String phone;
    private String street;
    private String city;
    private String state;
    private String zipCode;
    private String country;
    private double baseSalary;
    private double bonus;
    private double stockOptions;
    private String bankName;
    private String bankAccountNumber;
    private String bankRoutingNumber;
    private String taxId;
    private double federalTaxRate;
    private double stateTaxRate;
    private List<String> skills;
    private String department;
    private String managerId;
    private LocalDate hireDate;
    private List<PerformanceReview> reviews;

    public String getFullAddress() {
        return street + ", " + city + ", " + state + " " + zipCode + ", " + country;
    }

    public boolean isInState(String stateCode) {
        return this.state.equals(stateCode);
    }

    public double calculateGrossPay() {
        return baseSalary + bonus + stockOptions;
    }

    public double calculateFederalTax() {
        return calculateGrossPay() * federalTaxRate;
    }

    public double calculateStateTax() {
        return calculateGrossPay() * stateTaxRate;
    }

    public double calculateNetPay() {
        return calculateGrossPay() - calculateFederalTax() - calculateStateTax();
    }

    public String generatePayStub() {
        return "Pay Stub for " + name + "\n"
             + "Gross: $" + calculateGrossPay() + "\n"
             + "Federal Tax: $" + calculateFederalTax() + "\n"
             + "State Tax: $" + calculateStateTax() + "\n"
             + "Net: $" + calculateNetPay();
    }

    public void processDirectDeposit() {
        // connects to bank API using bankName, bankAccountNumber, bankRoutingNumber
        // transfers calculateNetPay() to employee's account
    }

    public double getAverageReviewScore() {
        return reviews.stream()
            .mapToDouble(PerformanceReview::getScore)
            .average().orElse(0);
    }

    public boolean isEligibleForPromotion() {
        return getAverageReviewScore() > 4.0
            && reviews.size() >= 3
            && skills.size() >= 5;
    }

    public String generatePerformanceReport() {
        StringBuilder sb = new StringBuilder();
        sb.append("Performance Report: ").append(name).append("\n");
        for (PerformanceReview review : reviews) {
            sb.append("  ").append(review.getDate())
              .append(": ").append(review.getScore())
              .append(" - ").append(review.getComments()).append("\n");
        }
        sb.append("Average: ").append(getAverageReviewScore()).append("\n");
        sb.append("Promotion Eligible: ").append(isEligibleForPromotion());
        return sb.toString();
    }

    // ... dozens more methods for benefits, time-off, training, etc.
}

This Employee class manages identity, address, compensation, tax calculation, payroll processing, banking, performance reviews, and promotion logic — at least six or seven distinct responsibilities.

What principle does it break?

Single Responsibility Principle (SRP). A class should have only one reason to change. This Employee changes when address formatting rules change, when tax law changes, when the bank API changes, when performance review criteria change, when payroll requirements change — and so on. Each of those is a separate axis of change, yet they're all coupled into one class.

It also violates high cohesion: the fields and methods don't form a tight, related group. The bank routing number has nothing to do with performance reviews, yet they sit side by side. Low cohesion is the hallmark of a class that's trying to be everything to everyone.

How to fix it

The primary refactoring is Extract Class: identify clusters of fields and methods that belong together and move them into their own classes.

public class Address {
    private String street;
    private String city;
    private String state;
    private String zipCode;
    private String country;

    public String getFullAddress() {
        return street + ", " + city + ", " + state + " " + zipCode + ", " + country;
    }

    public boolean isInState(String stateCode) {
        return this.state.equals(stateCode);
    }
}

public class Compensation {
    private double baseSalary;
    private double bonus;
    private double stockOptions;

    public double calculateGrossPay() {
        return baseSalary + bonus + stockOptions;
    }
}

public class TaxCalculator {
    private double federalTaxRate;
    private double stateTaxRate;

    public double calculateFederalTax(double grossPay) {
        return grossPay * federalTaxRate;
    }

    public double calculateStateTax(double grossPay) {
        return grossPay * stateTaxRate;
    }

    public double calculateTotalTax(double grossPay) {
        return calculateFederalTax(grossPay) + calculateStateTax(grossPay);
    }
}

public class BankAccount {
    private String bankName;
    private String accountNumber;
    private String routingNumber;

    public void processDirectDeposit(double amount) {
        // connects to bank API and transfers amount
    }
}

public class PerformanceRecord {
    private List<PerformanceReview> reviews;

    public double getAverageScore() {
        return reviews.stream()
            .mapToDouble(PerformanceReview::getScore)
            .average().orElse(0);
    }

    public boolean isEligibleForPromotion(int requiredReviews,
                                           double minScore,
                                           int minSkills,
                                           List<String> skills) {
        return getAverageScore() > minScore
            && reviews.size() >= requiredReviews
            && skills.size() >= minSkills;
    }

    public String generateReport(String employeeName) {
        StringBuilder sb = new StringBuilder();
        sb.append("Performance Report: ").append(employeeName).append("\n");
        for (PerformanceReview review : reviews) {
            sb.append("  ").append(review.getDate())
              .append(": ").append(review.getScore())
              .append(" - ").append(review.getComments()).append("\n");
        }
        sb.append("Average: ").append(getAverageScore());
        return sb.toString();
    }
}

public class Employee {
    private String name;
    private String email;
    private String phone;
    private String taxId;
    private String department;
    private String managerId;
    private LocalDate hireDate;
    private List<String> skills;

    private Address address;
    private Compensation compensation;
    private TaxCalculator taxCalculator;
    private BankAccount bankAccount;
    private PerformanceRecord performanceRecord;

    public double calculateNetPay() {
        double gross = compensation.calculateGrossPay();
        double tax = taxCalculator.calculateTotalTax(gross);
        return gross - tax;
    }

    public void processPayroll() {
        bankAccount.processDirectDeposit(calculateNetPay());
    }
}

Employee is now a thin coordinator. Each extracted class is cohesive (its fields and methods are tightly related), independently testable, and changes for exactly one reason.

Supporting refactorings

Extract Subclass

When some fields and methods are only used in specific variants of the class, pull them into a subclass rather than a sibling class.

Before — Employee has fields that only apply to certain types:

public class Employee {
    private String name;
    private double baseSalary;
    private double hourlyRate;      // only for contractors
    private int hoursWorked;        // only for contractors
    private double bonus;           // only for full-time
    private double stockOptions;    // only for full-time

    public double calculatePay() {
        if (hourlyRate > 0) {
            return hourlyRate * hoursWorked;
        }
        return baseSalary + bonus + stockOptions;
    }
}

After — variant-specific fields live in subclasses:

public abstract class Employee {
    private String name;
    protected double baseSalary;

    public abstract double calculatePay();
}

public class FullTimeEmployee extends Employee {
    private double bonus;
    private double stockOptions;

    @Override
    public double calculatePay() {
        return baseSalary + bonus + stockOptions;
    }
}

public class Contractor extends Employee {
    private double hourlyRate;
    private int hoursWorked;

    @Override
    public double calculatePay() {
        return hourlyRate * hoursWorked;
    }
}

Extract Interface

When different clients use different subsets of a large class, define interfaces so each client depends only on what it needs. This makes the class's responsibilities visible and enforces boundaries.

Before — callers depend on the entire Employee class even when they only need a slice:

public class PayrollService {
    public void process(Employee employee) {
        // only needs pay calculation — but sees everything
        double net = employee.calculateNetPay();
        employee.processPayroll();
    }
}

public class DirectoryService {
    public String lookup(Employee employee) {
        // only needs contact info — but sees everything
        return employee.getName() + " - " + employee.getEmail();
    }
}

After — narrow interfaces expose only what each client needs:

public interface Payable {
    double calculateNetPay();
    void processPayroll();
}

public interface ContactInfo {
    String getName();
    String getEmail();
    String getPhone();
}

public class Employee implements Payable, ContactInfo {
    // ... full implementation
}

public class PayrollService {
    public void process(Payable payable) {
        double net = payable.calculateNetPay();
        payable.processPayroll();
    }
}

public class DirectoryService {
    public String lookup(ContactInfo contact) {
        return contact.getName() + " - " + contact.getEmail();
    }
}

This follows the Interface Segregation Principle (ISP): no client should be forced to depend on methods it doesn't use. It also makes it much clearer where the class should eventually be split.
