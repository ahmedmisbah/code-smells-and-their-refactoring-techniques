Global Data

Description

Data that can be read and modified from anywhere in the codebase — global variables, public static fields, singletons with mutable state, or any shared state with no access control. The problem isn't that the data exists; it's that any code, anywhere, can change it without the rest of the system knowing. When something goes wrong, the debugging question is no longer "what method changed this?" — it's "which of the 400 classes that can see this variable changed it, and in what order?"

Global data is seductive because it's easy. Need a value everywhere? Make it public static. Need configuration? Stuff it in a global map. Need a database connection? Singleton. Each decision feels harmless in isolation, but together they create invisible coupling — every piece of code that touches the global is silently coupled to every other piece, with no interface or contract mediating the relationship.

Fowler introduced this smell in his second edition as a distinct concern, separate from (but related to) Large Class and Feature Envy, because modern codebases have grown large enough that uncontrolled global state is one of the most persistent sources of hard-to-reproduce bugs.

Example

public class AppConfig {
    public static String databaseUrl = "jdbc:mysql://localhost:3306/prod";
    public static int maxRetries = 3;
    public static boolean debugMode = false;
    public static String currentUserId = null;
    public static Map<String, Object> cache = new HashMap<>();
}
public class OrderService {
    public void placeOrder(Order order) {
        if (AppConfig.debugMode) {
            System.out.println("Placing order: " + order.getId());
        }
        // uses AppConfig.databaseUrl to connect
        // uses AppConfig.maxRetries for retry logic
        // writes to AppConfig.cache
    }
}

public class ReportService {
    public void generateReport() {
        // reads AppConfig.currentUserId — who set this? when?
        // reads AppConfig.cache — is the data stale?
        // changes AppConfig.debugMode for verbose logging during report generation
        AppConfig.debugMode = true;
        // ... generate report ...
        AppConfig.debugMode = false;
        // if an exception fires between these two lines, debugMode stays true forever
    }
}

public class AuthService {
    public void login(String userId) {
        AppConfig.currentUserId = userId;
        // every class in the system now silently "knows" who's logged in
    }

    public void logout() {
        AppConfig.currentUserId = null;
        AppConfig.cache.clear(); // wipes cache for ALL services, not just auth
    }
}

Any class can read or write any field at any time. ReportService temporarily flips debugMode — but if it throws, the flag stays flipped. AuthService.logout() clears the entire cache, blindly affecting OrderService. currentUserId is set in one place and read in dozens, with no guarantee it's set before it's read. Testing any of these services in isolation is nearly impossible because they all depend on — and mutate — shared global state.

What principle does it break?

Encapsulation. Global data is the antithesis of encapsulation — there is no boundary, no controlled access, no contract. Any code can change it, which means no code can trust it. You cannot reason locally about behavior because any method anywhere might have silently changed the state you depend on.

It also undermines testability: you can't test a class in isolation if it reads and writes shared mutable state. Tests become order-dependent, flaky, and hard to parallelize.

How to fix it

The primary refactoring is Encapsulate Variable: wrap the global data behind methods that control access, then progressively restrict who can call those methods.

Step 1 — wrap the raw fields behind getters/setters:

public class AppConfig {
    private static String databaseUrl = "jdbc:mysql://localhost:3306/prod";
    private static int maxRetries = 3;
    private static boolean debugMode = false;

    public static String getDatabaseUrl() { return databaseUrl; }
    public static int getMaxRetries() { return maxRetries; }
    public static boolean isDebugMode() { return debugMode; }

    public static void setDebugMode(boolean mode) {
        debugMode = mode;
    }
}

This alone is a small improvement — you now have a single place to add logging, validation, or breakpoints when the value changes. But the data is still effectively global.

Step 2 — replace the static singleton with an injected dependency:

public class AppConfig {
    private final String databaseUrl;
    private final int maxRetries;
    private final boolean debugMode;

    public AppConfig(String databaseUrl, int maxRetries, boolean debugMode) {
        this.databaseUrl = databaseUrl;
        this.maxRetries = maxRetries;
        this.debugMode = debugMode;
    }

    public String getDatabaseUrl() { return databaseUrl; }
    public int getMaxRetries() { return maxRetries; }
    public boolean isDebugMode() { return debugMode; }
}
public class OrderService {
    private final AppConfig config;

    public OrderService(AppConfig config) {
        this.config = config;
    }

    public void placeOrder(Order order) {
        if (config.isDebugMode()) {
            System.out.println("Placing order: " + order.getId());
        }
        // ...
    }
}

Now AppConfig is immutable. Each service receives its configuration through the constructor — explicit, testable, and impossible to silently mutate from another class.

Step 3 — split unrelated concerns into separate objects:

public class DatabaseConfig {
    private final String url;
    private final int maxRetries;
    // constructor, getters...
}

public class UserSession {
    private String currentUserId;

    public void login(String userId) { this.currentUserId = userId; }
    public void logout() { this.currentUserId = null; }
    public String getCurrentUserId() { return currentUserId; }
}

public class CacheManager {
    private final Map<String, Object> cache = new HashMap<>();

    public void put(String key, Object value) { cache.put(key, value); }
    public Object get(String key) { return cache.get(key); }
    public void clearFor(String namespace) {
        cache.keySet().removeIf(k -> k.startsWith(namespace + ":"));
    }
}

Each service receives only what it needs:

public class OrderService {
    private final DatabaseConfig dbConfig;
    private final CacheManager cache;

    public OrderService(DatabaseConfig dbConfig, CacheManager cache) {
        this.dbConfig = dbConfig;
        this.cache = cache;
    }
    // ...
}

public class AuthService {
    private final UserSession session;
    private final CacheManager cache;

    public AuthService(UserSession session, CacheManager cache) {
        this.session = session;
        this.cache = cache;
    }

    public void logout() {
        session.logout();
        cache.clearFor("auth"); // only clears auth-related cache
    }
}

The god-object AppConfig is gone. Each dependency is explicit, scoped, and testable. AuthService.logout() can no longer accidentally wipe OrderService's cache. And you can test OrderService by injecting a mock DatabaseConfig without touching any global state.
