Mutable Data

Description

Data that can be changed after it's created — and is changed, often from multiple places, often unexpectedly. Mutable Data is the close companion of Global Data, but it's broader: even data that isn't globally accessible can cause problems if it's mutable and passed around. You hand an object to a method, that method modifies it, and now the caller's state has changed out from under it. Or two threads share a mutable list, and one adds an element while the other is iterating.

The core issue isn't that mutation exists — some mutation is unavoidable and natural. The problem is uncontrolled, widespread mutation: when data can be changed from many places, it becomes impossible to reason about what state the data is in at any given point in the program. Every method that receives a mutable object becomes a potential source of surprise.

Fowler added this as a distinct smell in the second edition, influenced by the functional programming insight that immutable data eliminates entire categories of bugs.

Example

public class ShoppingCart {
    private List<CartItem> items;
    private double discount;

    public ShoppingCart() {
        this.items = new ArrayList<>();
        this.discount = 0;
    }

    public List<CartItem> getItems() {
        return items; // returns the actual internal list — anyone can mutate it
    }

    public void setDiscount(double discount) {
        this.discount = discount;
    }

    public double getTotal() {
        double sum = 0;
        for (CartItem item : items) {
            sum += item.getPrice() * item.getQuantity();
        }
        return sum - discount;
    }
}
public class CheckoutService {
    public void checkout(ShoppingCart cart) {
        List<CartItem> items = cart.getItems();
        items.removeIf(item -> !item.isInStock()); // silently modifies the cart
        double total = cart.getTotal();
        // ...
    }
}

public class PromotionService {
    public void applyPromotion(ShoppingCart cart) {
        cart.setDiscount(20.0);
        cart.getItems().add(new CartItem("Free Gift", 0, 1)); // injects an item
    }
}

public class AuditService {
    public void logCart(ShoppingCart cart) {
        List<CartItem> items = cart.getItems();
        // just reading... but another thread calls checkout() concurrently
        for (CartItem item : items) { // ConcurrentModificationException possible
            log(item);
        }
    }
}

CheckoutService removes items through the returned list reference. PromotionService injects items the same way. AuditService thinks it's just reading, but it's sharing the same mutable list that other services are modifying. The discount can be set from anywhere at any time. The cart's state is unpredictable.

What principle does it break?

Encapsulation and least surprise. When a getter returns a mutable internal collection, the class has lost control of its own state. Callers can bypass any validation, invariant checking, or event notification the class might want to enforce. The object's state changes without the object's knowledge.

It also violates defensive programming: the system provides no protection against accidental or malicious modification, making it fragile and hard to debug.

How to fix it

Return defensive copies or unmodifiable views

The simplest first step: stop exposing mutable internals.

public class ShoppingCart {
    private final List<CartItem> items = new ArrayList<>();
    private double discount;

    public List<CartItem> getItems() {
        return Collections.unmodifiableList(items); // read-only view
    }

    public void addItem(CartItem item) {
        if (item.getQuantity() <= 0) {
            throw new IllegalArgumentException("Quantity must be positive");
        }
        items.add(item);
    }

    public void removeOutOfStock() {
        items.removeIf(item -> !item.isInStock());
    }

    public void applyDiscount(double discount) {
        if (discount < 0) throw new IllegalArgumentException("Negative discount");
        this.discount = discount;
    }

    public double getTotal() {
        double sum = items.stream()
            .mapToDouble(i -> i.getPrice() * i.getQuantity())
            .sum();
        return sum - discount;
    }
}

Now callers can't add or remove items through the getter — they must use addItem and removeOutOfStock, which the cart controls.

Use immutable value objects

For data that represents a value (money, a date range, a coordinate), make it immutable from birth. Changes produce new instances instead of modifying existing ones.

Before — a mutable price that gets changed in place:

public class Price {
    private double amount;
    private String currency;

    public void setAmount(double amount) { this.amount = amount; }
    public void setCurrency(String currency) { this.currency = currency; }
}

After — an immutable value; "changes" return new objects:

public final class Price {
    private final double amount;
    private final String currency;

    public Price(double amount, String currency) {
        this.amount = amount;
        this.currency = currency;
    }

    public Price withAmount(double newAmount) {
        return new Price(newAmount, this.currency);
    }

    public Price add(Price other) {
        if (!this.currency.equals(other.currency)) {
            throw new IllegalArgumentException("Currency mismatch");
        }
        return new Price(this.amount + other.amount, this.currency);
    }

    public double getAmount() { return amount; }
    public String getCurrency() { return currency; }
}

No one can silently change a Price after it's created. If you need a different price, you get a new object — the old one remains intact for anyone still referencing it.

Separate queries from commands

A method should either return a value (query) or change state (command), not both. This makes mutation visible and predictable.

Before — a method that both mutates and returns:

public class Stack<T> {
    private final List<T> elements = new ArrayList<>();

    public T pop() {
        T top = elements.remove(elements.size() - 1); // mutates AND returns
        return top;
    }
}

After — separate the query from the command:

public class Stack<T> {
    private final List<T> elements = new ArrayList<>();

    public T peek() {        // query: returns value, no mutation
        return elements.get(elements.size() - 1);
    }

    public void pop() {      // command: mutates, returns nothing
        elements.remove(elements.size() - 1);
    }
}

The caller now explicitly chooses to observe (peek) or to mutate (pop). The mutation is always deliberate, never a side effect of reading.
